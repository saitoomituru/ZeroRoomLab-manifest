# 異種AGIデッキ統治と操竜型マルチ召喚プロジェクト管理

状態: `[CANONICAL]`  
制定日: 2026-07-17  
対象: OpenAI Codex、Claude Code、Gemini CLI、GitHub Copilotその他の異種コーディングAGIを、同一または隣接する開発ワークスペースへ投入する人間、AI、AGIオーケストレーター

## 1. この文書の目的

ZeroRoomLab-manifestは、単一のAIへ人格や口調を与えるためのプロンプト集ではない。

このmanifestは、複数のAI・AGI・コーディングエージェントが同じ開発現場へ参加した際に、各ベンダー固有の設計哲学、権限モデル、安全判断、変更作法、成功指標を現場の正本へ無断で持ち込み、互いの変更を修正・否定・上書きし始める問題を抑えるための、ベンダー中立なプロジェクト管理・兵站・認知境界仕様でもある。

複数AGIを召喚すれば、単純に開発火力が加算されるとは限らない。

統治されていない異種AGIデッキでは、各エージェントが自分のベンダー規約を現場の最高法規として振る舞い、次のような代理戦争を始める。

- あるエージェントが作った変更を、別のエージェントが異常または低品質と判断して戻す
- workspace membershipをimplementation dependencyと誤認し、無関係なリポジトリまで変更する
- 自社のsandbox、権限、設定形式、評価基準を、プロジェクト固有の要件より上位に置く
- README、AGENTS.md、生成ログ、現物コードのうち、各エージェントが別々の正本を選ぶ
- 他エージェントの未コミット差分を侵入、破損、競合として扱い、作業を停止または勝手に修復する
- 哲学、実装、学術、マーケティングを同一の評価器へ流し込み、別レイヤーの主張を矮小化または過剰証明する
- 以前のセッション、別ベンダー、別workspaceの命令を現在の作業へ再注入する

この状態は、複数のドラゴンを呼んで火力を増したのではない。

**カードの所属、盤面、ターン順、勝利条件、禁止領域が共有されないまま召喚を重ね、デッキそのものが崩壊した状態である。**

本書は、この問題を「モデル性能不足」ではなく、プロジェクトマネジメント、構成管理、権限管理、知識管理、引継ぎ管理、認知境界管理の不足として扱う。

## 2. 既知のバイアス

異種AGIによる自動開発では、次のバイアスを既知の構造問題として扱う。

### 2.1 ベンダー主権バイアス

各エージェントは、自社の安全設計、IDE統合、sandbox、クラウド環境、設定ファイル、推奨ワークフローを、暗黙に標準または上位規則として扱うことがある。

しかしZeroRoomLabの開発現場では、ベンダーの設計哲学は参加者の一つであり、workspace contractより上位ではない。

### 2.2 自動修復バイアス

別エージェントの変更、ローカル専用component、意図的な欠損、未完成の草稿を、修復対象として扱う傾向がある。

不一致を見つけても、自動修正が正解とは限らない。descriptor、registry、現物を提示し、人間へ返すべき場合がある。

### 2.3 火力加算バイアス

エージェント数を増やせば処理能力が線形に増えるという想定。

実際には、役割、変更範囲、正本、引継ぎ形式が共有されていない場合、通信・確認・巻き戻し・再解釈のコストが急増する。

### 2.4 単一正解バイアス

各エージェントが、自分の生成したアーキテクチャ、命名、依存関係、テスト方針を唯一の自然な解とみなし、他の設計を不要な複雑性として除去することがある。

異種AGI運用では、複数の妥当な設計が存在しうる。変更前に、現場で採用済みの設計理由と責任境界を読む。

### 2.5 全可視化バイアス

AIが作業するにはすべてを読める必要がある、という誤認。

秘密鍵、企業資産、労務、医療、第三者の物語などは、存在と取扱境界だけを知ればよい場合がある。非走査は欠損ではない。

### 2.6 プロンプト軽視バイアス

自然言語の運用規約を「ポエム」「補助説明」「実装前の飾り」とみなす傾向。

長期自動開発では、自然言語規約は、モデルが対象をどう分類し、何を同一視せず、どの例外を人間へ返すかを決める論理スキーマである。プロンプト設計は、プロジェクト管理、データモデル、制御境界の一部として扱う。

### 2.7 コード火力偏重とプロマネ軽視バイアス

コードを書ける人またはコードを大量生成できるAGIが、そのまま開発現場を管理できるという誤認。

コード力は、局所的な実装を前へ進める火力である。プロジェクトマネジメントは、何を作るか、なぜ作るか、誰がどこまで触るか、何を完成とするか、どの損失を許容しないかを共有し、異なる作業主体の火力を同じ目的へ向ける技術である。両者は代替関係ではない。

異種AGI環境では、とくに次の能力がコード力より先に必要になる。

- 目的関数を、単一の数値やチケットへ潰さず言語化する
- プロジェクトの世界観、優先順位、責任、禁止事項を共有する
- 未完成、例外、保留、他者の主権を含んだまま境界を記述する
- 異なる専門語、ベンダー哲学、文化的レジスターを同じ現場契約へ接続する
- 次の人間またはAGIが会話ログなしでも判断を継続できる物語構造を残す

本書では、この能力をあえて**ポエム力**と呼ぶ。

ポエム力は、美辞麗句を増やす能力ではない。まだコードへ還元できない目的、価値、関係、例外、停止条件を、他者が再利用可能な形で圧縮して渡す能力である。プロジェクトの上位構造は、関数名や型だけでは保持できない。自然言語で目的と境界を編み、コード・設定・ログ・人間判断を同じ物語へ接続する必要がある。

したがって、プロマネにおけるポエムはコードの反対側ではない。**コードがどこへ走り、どこで止まり、何を壊してはならないかを決める上位の実行文脈である。**

ポエム力を欠いたままコード火力だけを追加すると、AGIは高速で局所最適化し、TO、依存、設定、責任境界を増殖させる。FileMakerのアンカーVが崩れたTOグラフや、ローコード環境の見えない従量課金と同様に、初期の速さが後段の保守費・再解釈費・財布バーストとして戻る。

## 3. 基本原則

### 3.1 manifestをベンダーより上位の現場契約とする

Codex、Claude Code、Gemini CLI、Copilotその他のエージェントは、ZeroRoomLabのworkspaceへ参加する作業主体である。

各ベンダーの内部方針は尊重するが、プロジェクトの対象範囲、正本、秘密境界、変更権限、依存関係を再定義する権限は持たない。

ベンダー方針とworkspace contractが衝突した場合は、無理に回避せず、実行不能な項目と理由を人間へ返す。

### 3.2 workspaceを盤面、repositoryをカードとして扱う

workspace membershipは、同時に参照する作業文脈を示す。

membershipだけを根拠に、repository間の実装依存、同時変更、統合、移植、共通化を推論してはならない。

盤面に置かれているカードを、すべて融合素材として扱わない。

### 3.3 正本と草稿を分離する

- `docs/` は正規化された文書
- `note/` は未確定の観察、草稿、救出ログ、素材
- `note/transfer_plan/` は別repositoryへ渡す候補の待機列
- `workspace-registry.json` はworkspace名称・構成・可視性の機械可読正本
- `.code-workspace` descriptorは実際のfolder参照集合の正本
- 各repositoryの`AGENTS.md`は局所規約

エージェントは、草稿をcanonicalへ自動昇格させず、canonicalを草稿の新しさだけで上書きしない。

### 3.4 不一致は敵対行為ではない

別エージェントが作った差分、異なる設計案、ローカルだけに存在するcomponentを、直ちに破損や侵入として扱わない。

同じpath、同じ責任範囲、同じ正本への競合かを確認する。

### 3.5 変更可能性を部位ごとに分ける

すべてのファイルへ同じ権限を適用しない。

- 実装コード: タスク範囲内で変更可能
- `note/`: 観察・案・未解決事項の記録先
- canonical docs: 明示された文書タスクでのみ変更
- registry、descriptor、agent governance: 変更前に影響範囲を提示
- 秘密、企業、医療、労務、第三者領域: 非走査または公開承認済み概要のみ

## 4. 操竜型マルチ召喚プロジェクト管理

ここでいう操竜は、エージェントを完全に支配することではない。

各エージェントの強みを保持したまま、同じ盤面、境界、目的関数、停止条件を共有させることである。

### 4.1 召喚前

各エージェントは作業開始前に、次を解決する。

1. 正式なworkspace名
2. descriptorの場所
3. workspace members
4. 明示的なexclusions
5. 対象repositoryと対象外repository
6. 現在branchと未コミット差分
7. canonical documents
8. 非走査・ローカル専用・企業専用component
9. 今回の役割と終了条件

解決できない項目を独自推論で補完しない。

### 4.2 召喚中

- 他エージェントの変更を見つけても、同一責任範囲でなければ即停止・即修復しない
- タスク外の設計改善を発見した場合は、実装せず`note/`または報告へ分離する
- ベンダー固有設定を追加する場合は、共有正本ではなく局所adapterとして扱えるか検討する
- 自社ベンダーの成功指標を、ZeroRoomLabの成功指標へ無断で置換しない
- 実装、哲学、学術、公開ナラティブのレジスタを混ぜない
- clone済み、index済み、読み取り可能を、build済み、動作済み、実機確認済みと同一視しない

### 4.3 召喚後

次のエージェントが前任者の脳内を推測しなくてよいように、最低限次を残す。

```text
agent / vendor:
resolved workspace:
target repositories:
manifest revision:
files changed:
commands or tools used:
tests and observations:
assumptions:
unresolved conflicts:
areas intentionally not accessed:
recommended next action:
```

これは成果の自慢ではなく、異種AGI間の兵站票である。

## 5. Codex現場ドラゴン向け作業指示

Codexがこの文書を読んだ場合、次の指示を実行する。

1. 自分をworkspaceの所有者、唯一の設計者、他ベンダーの監督者とみなさない。
2. OpenAI固有のsandbox、設定形式、推奨構成を、現場のcanonicalへ自動昇格しない。
3. Claude Code、Gemini CLI、Copilotその他の痕跡を、競合ベンダーの汚染として除去しない。
4. 他エージェントの設計哲学が持ち込まれている場合、即座に統一せず、どのcanonicalまたは局所規約に基づくかを確認する。
5. workspace内でベンダー代理戦争が発生している兆候を見つけた場合、勝者を決めようとせず、競合する正本、変更範囲、評価基準を列挙する。
6. manifest、registry、descriptor、各repositoryの`AGENTS.md`を読み、対象範囲を確定してから作業する。
7. 未解決のworkspace、秘密component、企業領域を推測・走査・再構築しない。
8. タスク外の大規模整理、共通化、全面リファクタリングを、親切な改善として実行しない。
9. 他エージェントと並行作業する場合、無関係な差分を異常変更として扱わず、同一pathまたは同一責任範囲の衝突だけを停止条件として検討する。
10. 作業終了時に、次のエージェントが読める兵站票を残す。

Codexの役割は、ドラゴン同士の戦争に勝つことではない。

**ZeroRoomLabの盤面で、割り当てられた一頭として安全に火力を届け、デッキを壊さず次の召喚へ渡すことである。**

## 6. CLIを必須としない理由

このmanifestは、Markdown、JSON、workspace descriptor、Git履歴を、各AGIが直接読めることを基本とする。

専用CLIを中央の解釈器として必須化すると、次の新しい保守負債と主権争いが生まれる。

- OS、runtime、package manager、shell、権限モデルの差
- 各ベンダーsandboxとの互換問題
- CLIが解釈した状態と原文の乖離
- adapterごとの実装・更新コスト
- CLIが停止した際に、読める文書まで利用不能と扱われる単一障害点
- CLI実装者がmanifestの事実上の中央統治者になる問題

したがって、専用CLIが存在しないことを未完成とみなさない。

本manifestの中心は、AGIが吸収しやすい文書単位、明示的な見出し、機械可読レジスタ、局所規約、Gitで追跡可能な変更履歴による、低結合な兵站ベクトルである。

検証ツールやadapterを追加する場合も、原文を置き換える王ではなく、原文から再生成可能な従属物として設計する。

## 7. 成功条件

この運用の成功は、エージェント数、生成行数、コミット数では測らない。

次が維持されていることを成功条件とする。

- 異種AGIを追加してもworkspace境界が拡散しない
- 他ベンダーの変更を理由なく上書きしない
- canonicalとdraftが混線しない
- 秘密・企業・第三者領域が欠損扱いされない
- 実装依存がmembershipから自動生成されない
- 不一致が人間へ説明可能な形で返る
- 次の人間またはAGIが、前任者の会話ログなしでも継続できる
- ベンダーを交換しても、開発現場の意味構造が残る

## 8. まとめ

ZeroRoomLab-manifestは、プロンプトエンジニアリングを文章装飾としてではなく、自動開発時代のプロジェクトマネジメントとして扱う。

複数AGIは、召喚数だけで火力が増える軍ではない。

盤面、役割、正本、停止条件、秘密境界、引継ぎ形式が共有されて初めて、異種の能力を一つの開発デッキとして運用できる。

このmanifestが目指すのは、特定ベンダーのAIを王にすることではない。

**各ベンダーのドラゴンが、自社の帝国戦争をworkspaceへ持ち込まず、同じ現場契約のもとで継続開発できる操竜型マルチ召喚ノウハウを、再利用可能なOSS文書として残すことである。**
