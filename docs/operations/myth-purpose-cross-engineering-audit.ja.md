# 神話・目的関数の横断工学監査規約

状態: `[CANONICAL]` `[Layer A/B bridge]`  
制定日: 2026-07-17  
対象: 安全工学、制度工学、金融工学、情報工学、人間工学、GUI、KPI設計、AIエージェント運用

## 1. 上位規則

神話、哲学、ロマン、宗教的背景、比喩は、工学に付着した交換可能なフレーバーではない。可能な範囲で、それらが何の目的関数、価値、禁止事項、停止条件、未来像を保持しているか確認する。

工学的実装と神話的目的は同一ではないが、片方を消去してよい関係でもない。

- 神話・哲学は、何を守り、何を得て、何をしてはならないかを保持する意味側の車輪
- 工学、資源、検証、制度、予算、UIは、その目的を現実へ壊さず写像する実行側の車輪

両輪のどちらかが欠けた場合、実装成功、制度適合、利益達成、KPI達成だけで成功と判定しない。

## 2. 「フレーバーだから関係ない」警報

次の処理は目的関数矮小化の候補として警報を立てる。

- 神話や比喩を装飾として削除する
- 宗教的・哲学的背景を実装要件と無関係と即断する
- KPIへ変換できない価値を対象外にする
- GUIを効率だけで評価し、利用者の主権、儀礼、意味、撤退権を消す
- 金融モデルで資金回収だけを残し、研究目的や贈与経済の構造を消す
- 制度適合のために制度が保護すべき元の目的を消す
- 情報圧縮の過程で、停止条件、例外、禁則、物語上の責務を落とす

ただし、すべての比喩をそのままコード、契約、会計項目へ入れる必要はない。実装へ直接写像しない場合でも、その比喩が担っていた目的関数を別フィールドで保持する。

### 2.1 商業KPIを真理定規へ昇格しない

市場規模、売上、スポンサー損失、企業採用数、スタッフ数、調達慣行、法務事故件数は、それぞれの運用環境で重要な観測値になりうる。しかし、金銭へ変換しやすいこと、採用者が多いこと、損失額が大きいことは、別のproduct、利用者、贈与経済、少数者、実験系に対する普遍的な正しさを証明しない。

特に、商業systemと非商業／演者主権systemでは同じfailureへ異なるKPIを置く場合がある。

| 観測するfailure | 商業系で支配的になりうるKPI | 演者／実験系で支配的になりうるKPI |
|---|---|---|
| source endpoint変更 | 誤接続、スポンサー損失、送出責任 | ショー復帰時間、操作中断、コンテンツ継続 |
| local bootstrap認証 | 組織監査、accountability、配布責任 | ジャンク機材の即時復旧、local onboarding、所有者主権 |
| hardware故障 | 保証、交換在庫、検品母集団 | 現物隔離、ガムテ復旧、投げ銭までの継続運用 |
| UI warning | operatorの責任確認、監査trace | 演者の認知負荷、片手操作、演技継続 |

Layer Aでは、この差を安全／危険、正しい／誤りという単一の技術scoreへ潰さない。どの損失を支配的と見て、どのriskへ逃がすかを分解し、対象productのdeclared purposeと利用者を確認する。

一方Layer BとREADME表紙では、projectが相容れない哲学を明示的に批判してよい。upstream実装への技術的敬意は、スポンサーの財布、企業責任、制度保全を演者の人生、表現、復帰より上位に置く思想への中立義務ではない。ZeroRoomLabは、資本側が多数であることを理由にその主人を引き継がない。

商業KPIと責任回避を求める利用者は、そのために作られた商業deviceを選べる。OSSの限られた資源で同じ優先順位を安価に複製する義務はない。open deviceを作る意味は、商業KPIが取りこぼす利用者、用途、失敗からの復帰、表現の主権へ別の目的関数を実装できる点にある。

upstreamの枯れたcomponentが異なるKPIを持つ場合、次の順で扱う。

1. upstreamの挙動、成立条件、守っている対象を事実として記録する
2. upstream作者の未確認な思想や悪意を捏造しない
3. 対象productの原初目的、利用者KPI、許容するriskを独立に記録する
4. 衝突する場合、原初目的を商業既定値へ縮約せず、adapter、watchdog、policy profile、local implementationで差分を吸収する
5. 商業互換が本流の目的でなければ、商業device、外部profile、forkへ境界を返す。親切さを理由にproject資源を既存市場の再生産へ吸わせない

市場で主流であることは、互換性、利用可能な部品、既知failure、保守知識の量を評価する材料にはなる。これは採用価値の証拠であり、目的関数の裁定権ではない。

## 3. 工学分野ごとの適用

### 3.1 安全工学

Riskだけでなく、守るべき未来、期待利益、原初目的、必要資源、安全率、撤退余裕を同時に確認する。神話は危険受容の燃料ではなく、安全率を要求する理由である。

### 3.2 制度工学

制度、規制、契約、申請、資格境界が、元来どの人、権利、資源、信頼、再参入可能性を保護するために存在するか確認する。制度遵守そのものを目的化しない。

### 3.3 金融工学

収益、キャッシュフロー、投資回収、リスク係数だけでなく、研究継続、贈与経済、公共財、選択可能性、探索余裕を目的関数として保持する。ロマンを無償労働の担保にせず、ロマンを維持する資源設計へ接続する。

### 3.4 情報工学・KPI設計

代理指標が原初目的を食べていないか、各変換stageで監査する。記録対象には最低限、original purpose、変更理由、削減内容、unknown、影響、差し戻し経路を含める。

### 3.5 人間工学・GUI設計

GUIは操作効率だけの薄皮ではない。利用者が現在、未来、危険、利益、責任、撤退、未確定を区別する認知面である。神話的presentationは警報を隠してはならず、警報も未来価値を消してはならない。

## 4. 情報子工学との相互監査

情報子工学は、同じ入力がobserver、routing、output、verifierによって異なる結果へ解決される過程を追跡する。この横断監査は、神話・目的関数が各変換でどのように保持、変形、削減されたかを監査対象に加える。

推奨する最小記録:

```text
ψ: original observation / myth / purpose / request
∇φ: institutional, financial, informational, ergonomic routing
λ: policy, KPI, GUI, contract, code, operational output
Q: evidence, purpose-integrity check, resource-integrity check, unknown, rollback
```

相互監査は双方向である。

1. 神話側から、実装が目的関数を矮小化していないか監査する
2. 工学側から、神話が実行条件、資源、安全率、証拠を代替していないか監査する
3. 情報子工学側から、どのroutingで意味が変質したかtraceする
4. 結果を人格scoreではなくoperationとsnapshotへ付ける

## 5. 判定行列

| 目的・神話 | 実装・資源 | 判定 |
|---|---|---|
| 保持 | 十分 | 相互写像候補。検証を継続する |
| 保持 | 不足 | 資源欠損。使命感で埋めない |
| 欠損 | 十分 | KPI・制度・GUIドリフト候補 |
| 欠損 | 不足 | collapse候補。目的と実装を上流へ戻す |
| UNKNOWN | 任意 | 推定配置せず確認要求とする |

## 6. 主張境界

本規約は、神話や哲学が個別の工学判断を自動的に正当化すると主張しない。また、すべての工学事故が目的関数矮小化から生じるとも主張しない。

主張するのは、目的・神話の保持状態と、実装・資源・検証状態を独立に観測し、その変換経路を相互監査できるようにする必要がある、という運用上の設計原則である。
