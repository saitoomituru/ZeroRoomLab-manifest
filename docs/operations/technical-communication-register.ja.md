# テクニカルコミュニケーション運用規約

状態: `[CANONICAL]`  
制定日: 2026-07-15  
適用範囲: ZeroRoomLab-manifest および参照先リポジトリのREADME、技術文書、実験ノート、公開発信

## 1. 目的

同じ事実でも、READMEの表紙、実験ノート、note.comの記事、Xの投稿、YouTubeの映像では伝達の役割が異なる。本規約は、媒体ごとの説明様式を混同して事実を弱めたり、逆に観測範囲を越えて主張を拡張したりする事故を防ぐ。

これは免責文を増やすための規約ではない。観測できた事実は観測できた強さのまま書き、観測していない範囲には伸ばさないための規約である。

## 2. 媒体別レジスター

| 媒体・位置 | 主な読者 | 適切なレジスター |
|---|---|---|
| README冒頭のおおむね40行 | 一般読者、一般ナード、検索・SNS・スクリーンショット経由の読者 | 公開用の表紙。短く、強く、ビジョンと観測結果を伝える。マーケティングおよび芸術表現の射程として扱う |
| READMEの境界文以降 | 大人のOSSパイレーツ | ハードウェア、コミット、ログ、試験条件、主張境界を読んで評価する技術レジスター |
| `docs/`、実験記録、技術レポート | 実装者、再現者、監査者 | 条件、手順、結果、失敗、射程を明記する工学レジスター |
| リポジトリ内の `note/` | 整理前の調査者・編集者 | 一次資料、草稿、サルベージ記録。正本とは限らないが、原文と文脈を保つ |
| note.com等の小説・ドキュメンタリー・漫画・ZINE配信SaaS | 一般読者、作品読者 | マーケティング、芸術、物語のレジスター |
| X、YouTube、その他の短文・映像媒体 | 流入読者、視聴者 | 報道、告知、エンターテインメントのレジスター。媒体のテンポと文脈を優先する |

`note/` はリポジトリ内ディレクトリ、note.com は外部配信サービスであり、同じ「note」でも運用上は別物とする。

READMEには、公開用の表紙から技術レジスターへ切り替わる位置に次の英文を置く。

> The README opens with the public-facing view. The technical notes below assume readers will inspect the hardware, commits, logs, test conditions, and claim boundaries before extending any result.

この一文以降では、読者にも証拠と境界を読む責任がある。冒頭の表紙を技術仕様書として過読せず、深い技術文書を宣伝コピーとして過読しない。

## 3. 日本語の暗黙射程と翻訳時の扱い

日本語の技術会話では、直前に示した実機、ログ、現場、時点を共有射程として「動いた」「再現した」「互換した」と言い切ることがある。この言い切りは、文脈を外した普遍命題や、全実装とのバイト一致を自動的に意味しない。

したがって編集者とAIは、次を守る。

- 日本語で成立している観測事実を、英語圏向けの過剰な留保で弱めない
- 翻訳時は主張を弱める代わりに、hardware、commit、workload、log、test boundaryへの参照を残す
- 日本語の省略を無条件に普遍化せず、英訳では必要な主語・対象・試験射程だけを補う
- 「完全互換ではないかもしれない」のような総論を毎段落へ追加しない
- 語り口を変えても、観測結果そのものの強度は変えない

日本語版が英語版の下位写本になるわけではない。日本語で形成された概念と観測記述を正本として扱い、英語は異なる暗黙文脈へ接続するための変換層とする場合がある。

### 3.1 日本語・イギリス英語・米語の語用論的な写像

本規約では英語を一枚岩として扱わない。ただし、本リポジトリはen-GB翻訳ファイルを運用せず、英語出力は明示がない限りen-USを対象とする。イギリス英語は、日本語の社会的含意を直訳したときに生じる差を説明する比較材料に留める。

コーディングAIによる日本語からen-USへの意訳手順、語用論的な等価表現、各リポジトリへの導線ルールは、[コーディングAI向け日本語意訳レジスタ](coding-ai-japanese-paraphrase-register.ja.md)を正本とする。

## 4. 技術文書における主張の境界

技術文書では、次の証拠列を可能な範囲で接続する。

```text
hardware / runtime
    → commit / version
    → command / workload
    → log / measurement
    → observed result
    → test boundary
```

この列が成立する範囲では、「動作した」「再現できた」「機能互換を観測した」と明確に書いてよい。事実を「可能性がある」「厳密には不明」と一律に弱めない。

一方、次は別の主張であり、個別の証拠なしに同一視しない。

- 観測した機能互換と、内部実装またはバイト列の同一性
- 特定構成・負荷での再現と、すべての環境での完全互換
- 動作確認と、暗号強度・耐攻撃性・セキュリティ監査の保証
- 非公開部分があることと、違法改変・隠蔽・不正アクセスがあること

再現者に必要な結論は次の形で閉じる。

> 記載した実機、コミット、負荷、ログの範囲で機能互換を観測した。それ以上の内部同一性や未試験条件は本記録の射程外である。追加確認が必要な場合は実機と原ログを確認すること。

## 5. セキュリティ・暗号強度の記述

機能が観測できたことは記録してよい。しかし、実装全体を公開できず、独立したレッドチームまたは同等の監査を通していない場合、暗号強度や耐攻撃性は保証対象にしない。

推奨表現:

> 記載した試験範囲では機能動作を観測した。暗号強度と耐攻撃性は、実装開示および独立監査を完了していないため保証対象外である。

これは観測事実を取り消す免責ではなく、機能再現とセキュリティ保証を異なる検証問題として分離する記述である。鍵、個人識別情報、攻撃に転用できる実装詳細を開示しないことも、直ちに不正の証拠にはならない。

## 6. 過剰防壁が生む逆方向の事故

善意から法務防戦的な留保を重ね、詳細を語らず、観測事実まで曖昧にすると、次の誤読へ燃料を与えることがある。

- 本当に違法な改変や隠蔽があるため説明できない
- マルウェア的な挙動差を意図的に覆い隠している
- 実際には誰でも簡単に再現でき、技術的蓄積がない
- 作品上の陰謀論・毒語りが、現実の隠蔽疑惑として読まれる

工学的には、監査可能な入出力と境界を示さずにラッパーや難読化だけを増やすと、正常な実装でもマルウェア的に見える問題に近い。対策は免責文の壁ではなく、公開可能な証拠列と試験境界を提示することである。

ただし、誤読を恐れて非公開情報を開示したり、未観測範囲まで証明したりする必要はない。示した証拠の範囲で閉じる。

## 7. 編集・AI運用ルール

- README冒頭、note.com、X、YouTube等のマーケティング・芸術レジスターを、技術編集者やAIが自動的に弱めない
- マーケティング文を直す権限は、明示的に依頼された場合または観測事実との直接矛盾がある場合に限る
- 技術文書では観測事実を明言し、同時に条件と未試験範囲を記録する
- 深い技術文書の留保をREADME冒頭へ逆流させない
- README冒頭の表現を、技術文書の証拠として代用しない
- 物語上の陰謀論、毒語り、キャラクター表現を、技術的な事実認定へ混入させない
- 非公開であることだけを根拠に、違法性、安全性、危険性、実装同一性を推測しない

事実との矛盾が見つかった場合は、語調を弱めるのではなく、対象、条件、時点、証拠を修正する。
