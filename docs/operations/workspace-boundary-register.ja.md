# VS Codeワークスペース境界レジスタ

状態: `[CANONICAL]`  
制定日: 2026-07-17  
対象: ZeroRoomLabのリポジトリを扱う人間、コーディングAI、AGIオーケストレーター

## 1. 目的

ZeroRoomLabでは、VS Codeワークスペースを単なるエディタの表示設定ではなく、複数リポジトリを一つの作業文脈へ束ねる**認知境界・作業境界**として使う。

リポジトリ、ワークスペース、プロジェクト、実行環境、実装依存は別の概念である。現在ディレクトリや隣接リポジトリからワークスペースを推測すると、抽象メタエンジン、ベアメタル層、企業資産の境界を越える事故が起きる。本レジスタは、その誤結合を防ぎつつ、公開OSS側の開発環境を再構築可能にする。

## 2. 用語と責務

| 単位 | 責務 |
|---|---|
| リポジトリ | Git履歴、コード、文書、ライセンス、変更権限の単位 |
| VS Codeワークスペース | 複数リポジトリを束ねる作業・認知・参照集合の単位 |
| プロジェクト | 達成目標と進行管理の単位。複数ワークスペースにまたがりうる |
| 実行環境 | OS、ホスト、SDK、Python、接続実機、環境変数、資格情報等の実行条件 |
| 実装依存 | import、リンク、ビルド、通信、データ交換等の技術的依存 |
| manifest | 全ワークスペースに共通するAI認知、用語、主張境界、再構築手順の基盤 |

同じワークスペースに入っていることは、相互の実装依存を意味しない。別ワークスペースに分離されていることも、歴史的・思想的な無関係を意味しない。ワークスペース分離は、暗黙依存を作らないための疎結合境界である。

## 3. `.code-workspace`の扱い

`.code-workspace`はJSONまたはJSONC形式の構成記述であり、`folders`、ワークスペース設定、tasks、launch、extensions、環境変数参照等を持ちうる。ただし、それ自体は実行環境ではない。

スコープ確定時は、実在する`.code-workspace`の`folders`を一次資料とする。manifestの機械可読レジスタは名前と意味を解決するための索引であり、端末固有の絶対パスや最新のフォルダー列挙を置き換えない。

## 4. AI／AGIの必須解決手順

1. ユーザーが指定したワークスペース名・ファイル名・大文字小文字を保持する。
2. [workspace-registry.json](workspace-registry.json)で登録名と明示的aliasだけを解決する。
3. 対応する`.code-workspace`を読み、`folders`から対象集合を確定する。
4. ZeroRoomLab-manifestのAGENTS.mdと運用レジスタを共通コンテキストとして読む。
5. 各ワークスペースメンバーのAGENTS.mdを読み、より局所的な制約を適用する。
6. `cwd`、隣接フォルダー、READMEのリンク、思想的系譜から対象を暗黙拡張しない。
7. 登録名を別リポジトリ名へ自動訂正しない。
8. 解決できない場合は推測せず`⊥`として報告し、対象を確認する。

走査や変更を始める前に、少なくとも次を明示する。

```text
Workspace: <登録名>
Descriptor: <読み取った.code-workspace>
Members: <foldersから解決した参照集合>
Excluded: <登録外・別境界・非公開領域>
Environment: <確認済みの実行環境。未確認ならunknown>
```

## 5. 現行ワークスペースの意味

### Deb800

x800系のベアメタル／現場実装層。撮影、物理演出、配信、編集、実機計測を束ねる。SphereOSやFAM等の抽象メタエンジン実装を暗黙依存させない。

### スフィア

SphereOS、FAM、AQC、IBD、Quantaril Cloud等の抽象メタエンジン層。x800系とは系譜やインターフェースを共有しうるが、同一の作業境界には置かない。

実運用ワークスペースに含まれる`スフィア独鈷書`は、一般的な設計ノートではない。スフィアOSがワールドデータの境界を認識するための、エドワーズ系暗号による暗号チェーンの秘密鍵構造体・秘密鍵束である。内容の走査、要約、ログ出力、Git登録、外部送信、配布、公開環境への複製を禁止する。

将来、キーチェーン仕様が確定した場合は、SDKを利用する各サードパーティが自分のワールド境界を保存する鍵を自分で生成するための独鈷書生成ライブラリを別途公開できる。現存する秘密鍵束そのものをテンプレートや配布物へ転用してはならない。

### NeeT_kitchen_ALCHEMIST

低資源の生活者が、家庭設備、公開知識、人間の手・目・鼻・口によるフィードバックを使って食べられる成果物を作り、再現知識を共有する人間側の生活兵站研究。最初の実装は、家庭用ポテトチップス製造実験の`OpenSourcePITETO`である。

スフィアの抽象メタエンジン層にも、Deb800の撮影・演出・配信・編集・計測層にも所属しない。思想や実験作法を参照しても、SphereOS、FAM、x800機器を料理のruntime依存へしない。AI、高価な計測器、雇用、事業化が先に揃わなくても、人間の身体と家庭設備だけで閉じる研究を第一級の成果として扱う。

詳細は[人間側の生活兵站研究](../projects/neet-kitchen-alchemist.ja.md)を参照する。

### 企業向け建築DX検証ワークスペース

株式会社寺徳と行う建築DX、CCUS、マルチエージェント運用の検証環境が存在する。会社概要と取り組みの公開可能な概要はmanifestに記録できるが、内部は相手企業の資産を含む。

現場労務、労基、医療情報、企業固有のGemini等の指示、Google Drive内の資料、認証情報、内部フォルダー構成は公開OSSの再構築対象に含めない。知見が一般化され、企業資産から分離された抽象OSSになるまでは、存在と公開可能な概要だけを扱う。

## 6. 境界を越えるときの規則

- 別ワークスペースのリポジトリを読む・変更する場合は、対象を明示してスコープを再確定する。
- manifest内のリンクは参照導線であり、作業権限や実装依存を自動的に付与しない。
- サードパーティ用ワークスペースは、ライセンス、秘密情報、認証、データ所有者、出力公開範囲を個別登録する。
- ローカル暗号鍵束は「再構築できない資料」ではなく「再構築・複製してはならない秘密鍵資産」として扱う。
- 公開OSSへ移す知見は、企業・個人資産から切り離し、出典、権利、再現条件を確認する。
- ワークスペース名のaliasは人間が明示登録したものだけを使う。
