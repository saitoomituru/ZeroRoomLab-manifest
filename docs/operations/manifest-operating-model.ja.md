# マニフェスト運用モデル

作成日: 2026-07-13

## 1. 目的

ZeroRoomLab-manifest は、単なる公開用の説明書ではなく、次の3層を一つのリポジトリで運用するための基盤である。

1. 生ノート・ドラフト・サルベージログを保存する層
2. 正規化済みの哲学・理論・プロジェクト文書を置く層
3. 他リポジトリへ転送する前の候補を管理する層

## 2. 役割分担

### 2.0 運用モデルは内容レイヤーではない

この運用モデルは、AGENTS.md の三層スコープ規約を置き換えるものではない。note / docs / transfer_plan は、文書の内容を別の意味論的レイヤーへ移すための区画ではなく、保存・整理・転送を管理するための運用枠組みである。

各文書の主たる内容は引き続き Layer A / Layer B / Layer C のいずれかに属し、運用ガイドはそれを支えるための配備ルールとして扱う。

### 2.1 note

- 未整理・未正規化・一次資料の保管場所
- 旧世代ノート、リサーチメモ、調査報告書、ドラフトを置く
- ここに置いてあるものは、すべて即座に正規化済みとは見なさない

### 2.2 docs

- 哲学、理論、プロジェクトの正式な文書を置く
- ここに置く文書は、既存の構造とリンク関係を持つことを前提にする
- 既存の README / AGENTS.md / docs 系文書を更新する際の正本候補とする

### 2.3 note/transfer_plan

- 他リポジトリに反映予定の内容を集める場所
- 反映先ごとに対象・優先度・状態を記録する
- 反映済みのものはステータスを更新し、元のノートとの対応関係を明示する

## 3. 運用ルール

- 生のノートは直ちに docs へ移さない
- 内容を他リポジトリへ送る前に、対象リポジトリに合わせて再構成する
- 同じ内容を note / docs / transfer_plan の複数箇所に重複して置かない
- 更新時は、ハブ文書と参照先の関係を明示する
- README、技術文書、外部配信媒体の説明様式は混同せず、[テクニカルコミュニケーション運用規約](technical-communication-register.ja.md)に従う

## 4. 哲学・神学・信仰の保全ルール

哲学・神学・信仰に関する内容は、技術実装や運用手順の説明に吸収してはならない。これらは独立した意味構造として保持する。

- Layer B の内容は、要約や転送の過程で技術化・平板化されないようにする
- 重要な哲学・神学・信仰文書は、原文・要旨・文脈を失わない形で docs 配下に保持する
- 転送候補として抽出する場合でも、元文の語彙・主旨・境界線を削らない
- 旧ノートや転送キューから再利用する場合は、元の文脈が見失われないように出典と保存先を明示する
- もし要約が必要な場合は、要約そのものと元文への参照を分けて管理する

## 5. 既存ノートからの再利用対象

過去ノートの中で、次の種類の内容は本運用モデルに再配置しやすい。

- マニフェストの役割説明
- 既存文書の構造整理
- 複数リポジトリにまたがる設計・運用メモ
- 古い表現・旧用語と現行語彙の対照

## 6. 次の実装候補

- 既存ノートからの抽出テンプレートを作る
- docs/operations 配下に、転送ガイド・運用ガイド・状態表を増やす
- README から運用モデルへの入口を増やす
