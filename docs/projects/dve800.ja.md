# DVE800 — Deep Video Engine 800

GitHub: [saitoomituru/DVE800](https://github.com/saitoomituru/DVE800)

800シリーズの**編集レイヤー**。DaVinci Resolve（無料版）をローカルAIで強化する、完全ローカル動作の動画編集自動化エンジン。

```
OND800  → 撮る（NDIマルチカメラ）
SAO800  → 流す（OBS配信母艦）

```

## なぜDVE800か

CapCutはスマホ・クラウド前提のネタ動画量産ツール。Adobeは広告代理店のSFW制作向け。DaVinci公式にAI自動化はない。

| | CapCut | Adobe | DVE800 |
|---|---|---|---|
| ローカル完結 | ✗ | ✗ | ✅ |
| NSFW素材 | BAN | グレー | ✅ |
| 日本語形態素 | 雑 | 雑 | FAMで特化 |
| ドメイン語彙 | 汎用のみ | 汎用のみ | 話者別FAM |
| DaVinci連携 | ✗ | ✗ | ✅ |
| 価格 | 無料/サブスク | 高額サブスク | OSS無料 |

自分のマシンでWhisperを動かせば、GPUベンダーの言論ポリシー・CapCutのクラウド審査・Adobeの規約・OpenAIのコンテンツポリシーはすべて関係なくなる。あなたのマシン、あなたの責任、あなたの言論。

## アーキテクチャ（8モジュール構成）

```
素材（映像・音声）
  → [1] Whisper Discovery   既存Whisperを環境検出
  → [2] Whisper Installer   なければ自動取得・設定
  → [3] AQC-Local           FAM多義性マップ生成・話者別ベクトルDB（ChromaDB）
  → Whisper-L                音声書き起こし（FAMコンテキスト注入済み）
  → [4] DaVinci API Bridge  タイムライン自動構築・SRTインポート
  → [5] MCP Server          PSYCHO-Py800MCP統合口
  → [6] Cockpit             FastAPI + ブラウザUI（審査ノート・最終決定）
  → [7] Compliance Review   審査フィードバックの正規化・ショットマッチング
  → [8] Mosaic Detection    外部検出モデル呼び出し（AGPL-3.0境界を隔離）
```

三層プラグイン構成（Adobe Animation Composerを技術UXの参照点としつつ、Animation Composerにない「重量OFX処理層」を追加）:

| 層 | 役割 | 実装形態 |
|---|---|---|
| コックピット層 | FAM文脈判断・審査ノート取り込み・粒度の最終決定 | ローカルFastAPI + ブラウザUI |
| 軽量プリセット層 | 字幕演出・トス・タイトル等 | Fusion macro（コンパイル不要） |
| 重量OFX層 | モザイク検出座標の読み込み、強度パラメータのリアルタイム露出 | C++ OFXバンドル（Gyroflowと同型設計） |

### FAM多義性マップ

話者のSNS投稿群から、語彙がどのジャンル・文脈で使われているかを抽出したJSON構造体。Whisper-Lの`initial_prompt`に動的注入することで、技術用語・NSFW語彙・地域固有名詞・造語を高精度で書き起こす。tiny/base/small/mediumはinitial_promptの後半を無視するため、Whisper-Largeモデルが必須。

## 担げるロマン砲

DVE800は超兵器を作らない。担げるロマン砲を作る。

NSFW領域のモザイク自動検出・音声補正等の要素技術はOSSコミュニティで急速に充実してきている（DeepMosaics, lada, RVC等）が、いずれも「モデルの出力が最終解」という設計思想であり、プラットフォーム審査・自主規制団体審査のような、人間の心象がタイムコード単位で局所的に返ってくる実運用フローには噛み合わない。

DVE800は検出モデルは借りるが、**強度の最終決定権は常にオペレーターの手元に残す**。演者・監督両方の現場経験に基づく一次情報からの逆設計であり、「完全自動は無理」という判断による。

## 実装状況（2026-07-02時点）

現在**設計段階シーズン**。エンドツーエンドで起動できる状態ではない。

| Module | 状態 |
|---|---|
| 1: whisper_discovery | ✅ 実装・テスト済み |
| 2: whisper_installer | ⛔ 未着手（設計のみ） |
| 3: aqc_local | 🔶 基本テスト済み・設計見直し検討中 |
| 4: dav_api | 🔶 コード実装済み・実機未検証 |
| 5: mcp_server | ⛔ 未着手 |
| 6: cockpit | ⛔ 未着手（設計のみ） |
| 7: compliance_review | 🔶 型定義・レジストリ骨格のみ |
| 8: mosaic_detection | ⛔ 未着手（ライセンス隔離方針のみ確定） |

外部OFX（`Gyroflow.ofx.bundle`）が無料版DaVinci Resolveでロードできることは実機確認済み（2026-07-02）。これはDVE800自体の動作保証ではない。

### FAM実装バリエーションの状況

DVE800のModule 3（`aqc_local`）が接続する先は、[FAM三系統](../theory/fam-overview.ja.md#6-fam三系統現状ステータス2026-07-02時点)のうち以下の2つ：

| バリエーション | 状態 | 補足 |
|---|---|---|
| FAM over AQC（フルレベル） | 🔶 過去に動作実績あり・現在は利用不可 | 過去のある環境で一度フル動作した実績があるが、現在の情勢とサプライ事情により同環境を再現できず利用不可。サルベージが必要（[aqc.ja.md](../theory/aqc.ja.md)参照） |
| FAM over MCP | 🔶 トイモデル状態 | 概念検証レベル。Module 5（`mcp_server`）はディレクトリ未作成で実装これから |

DVE800のFAM実装は**FAMoverMCPのトイモデル検証**であり、FAMoverAQCの実績とは区別して扱う。
