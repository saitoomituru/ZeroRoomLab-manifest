# ZeroRoomLab Translation Style Guide

**Texas English Style — Not Corporate, Not Academic.**

---

## 1. What This Is For

GitHub READMEs and docs get read by English-speaking engineers.
But academic English kills the voice. Corporate English kills it faster.

**Goal: sounds like a Texas garage hacker wrote it.**
Texture over formality. Declarations over explanations. Honest over polished.

---

## 2. Tone Rules

| NG (kills it) | OK (keeps it alive) |
|---|---|
| "This project aims to provide..." | "This thing does X. Period." |
| "It is worth noting that..." | "Heads up —" / "Fair warning:" |
| "In order to facilitate..." | "To get X done," |
| "Furthermore, it should be mentioned" | "Also —" / "One more thing:" |
| "The author proposes..." | "My take:" / "Here's the deal:" |
| "This framework enables users to..." | "You can now..." / "Lets you..." |
| Passive voice stacking | Active voice first |
| Noun-clause towers | Short sentences chained |

## 4. Context-Specific Register

- **README**: Casual. Explain it like to a friend.
- **spec / docs**: Slightly more formal. But zero corporate.
- **Code comments**: Most casual. One line max.

---

## 5. Post-Translation Checklist

```
□ Did you use "utilize"? → Change to "use"
□ Did you use "in order to"? → Drop to "to"
□ Any sentence running 3+ lines? → Break it
□ Three passive sentences in a row? → Flip to active
□ Any fixed terms translated against §3? → Revert
□ Too many sentences starting with "it is"? → Give them a subject
```

## 9. Misinfo Guard

Never write these — they're documented errors:

- "national champion" → write "placed in nationals" or "nationals qualifier"
- "Apple employee" → write "contract-based engagement" or "brought in on contract"
