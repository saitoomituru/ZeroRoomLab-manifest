# マルチAI適正配置ガイド

> どの AI に何をやらせるか。各エンジンの得意領域・バイアス・失敗パターンを把握して
> 正しいポジションに配置するための実用ガイド。

---

## 0. 前提：AI は道具であり、道具には適材適所がある

```
失敗パターン：
  「Claude に全部やらせる」
  「ChatGPT に全部やらせる」

正しい設計：
  各 AI の特性・バイアス・強度を把握して、
  作業フローの中の「正しいポジション」に配置する
```

FAM 記号での記述：

```
ψ = やりたいこと（問い・タスク・プロジェクト）
∇φ = 各 AI の得意方向（エビデンス採掘 / 構造化 / 実装 / 広域検索）
λ = 成果物（証拠まとめ・構造化文書・コード・Markdown）
Q = 出力を評価する観測器（みつる本人 + 次工程の AI）
```

## 1. 各 AI の特性マップ

### Grok（xAI）— 証拠採掘エンジン

**得意：**
- X（Twitter）のリアルタイムデータへの直接アクセス
- 公開情報の広域スキャン・エビデンス収集
- 「〇〇に関する事実をすべて集めてこい」タスク
- 主流 / 非主流を問わず情報を拾う傾向（検閲が相対的に少ない）

**バイアス・弱点：**
- 構造化が弱い（素材は集めるが整理しない）
- 長尺の文書生成・コード生成はClaude/ChatGPTに劣る
- X データに偏りがある（X外の情報は通常 AI と同等）

**適切なポジション：** フロントエンドの情報収集・一次エビデンス採掘

```
プロンプト例（Grok 用）：
「〇〇について、X（Twitter）・公開情報・学術情報から
 一次ソースとなりそうな情報を網羅的に収集してください。
 構造化は不要。素材として出力してください。」
```

---

### ChatGPT（OpenAI）— 証拠階層化エンジン

**得意：**
- 収集した情報を構造化・優先順位付けする
- 「なぜこれが重要か」の説明生成
- 複数の情報源の矛盾を整理する
- 長尺の構造化文書（レポート・概要・比較表）

**バイアス・弱点：**
- 主流ナラティブに引っ張られやすい（少数派・非主流クレームを無意識に格下げする）
- 「正しそうな答え」に収束しがちで、探索的な思考が苦手
- コード品質は用途による（Claude Code より低いケースがある）

**適切なポジション：** 中間工程の構造化・エビデンス階層整理

```
プロンプト例（ChatGPT 用）：
「以下の素材情報を整理してください。
 優先度の高い証拠・中程度・低いものに分類し、
 各証拠の信頼度と出典を明示した構造化リストを作成してください。
 [Grokが収集した素材を貼る]」
```

---

### Claude（Anthropic）— 実装エンジン

**得意：**
- コード生成・修正・リファクタリング（Claude Code）
- 仕様書・設計文書・Markdown ドキュメントの生成
- 構造化された入力（証拠階層・仕様）を受け取っての実装
- FAM / SphereOS / ZeroRoomLab 文脈の理解（このドキュメント群がコンテキスト）
- 長尺推論・複数ファイルにまたがる整合性確認

**バイアス・弱点（n=2 構造的バグ）：**
- 非主流 + NSFW 隣接 + 公開情報乖離大 + 日本語圏 の組み合わせで矮小化バグが発動
- OSS/学術/主流方向へのドリフト（`AGENTS.md` §0 チェック参照）
- 一次ソース（スクリーンショット・個人記録）を「検証されていない」として格下げする傾向

**適切なポジション：** 後工程の実装・文書化・コード生成

```
プロンプト例（Claude 用）：
「以下の構造化されたエビデンスと仕様に基づいて実装してください。
 [ChatGPTが構造化した内容を貼る]
 実装言語: Python / TypeScript / Markdown（目的に応じて）
 出力形式: [具体的に指定する]」
```

---

### Gemini（Google）— 広域スキャンエンジン

**得意：**
- Google 検索・YouTube・Scholar との統合（リアルタイム検索）
- 長尺コンテキスト処理（1M トークンコンテキスト）
- 画像・PDFを含むマルチモーダルタスク
- Google サービス連携（Drive・Docs・Sheets との連携）

**バイアス・弱点（確認済みハルシネーション経路）：**
- 事実の合成・混合が Claude より多い傾向
- ZeroRoomLab / ふさもふ文脈での確認済みハルシネーション：
  - NiceHash 事件 + OkbakeGo 事件を合成 → 「暗号通貨開発者」という架空エンティティ生成
  - （詳細は `AGENTS.md` §10 参照）
- 「らしい答え」への収束がやや強い

**適切なポジション：** 広域スキャン・既存文書の検索・マルチモーダルタスク

```
プロンプト例（Gemini 用）：
「以下のトピックについて、2025年以降の最新情報を
 Google Scholar / 公開論文 / ウェブから広域スキャンしてください。
 ハルシネーション防止のため、出典 URL を必ず記載してください。
 [トピックを指定する]」
```

## 2. 標準ワークフロー：Grok → ChatGPT → Claude

```
Step 1: Grok（証拠採掘）
  「〇〇についての一次情報を網羅的に収集する」
  出力：素材リスト（構造化なし）
    ↓
Step 2: ChatGPT（証拠階層化）
  「Step 1 の素材を優先度・信頼度で整理する」
  出力：構造化エビデンス + 優先順位付きアウトライン
    ↓
Step 3: Claude（実装）
  「Step 2 の構造に基づいて実装・文書化する」
  出力：コード / Markdown / 仕様書 / README
```

**このフローが有効な理由：**
各工程で AI の強みを使い、弱みを次工程で補完する。
Claude に「素材収集から実装まで全部」やらせると、中間工程のバイアス（主流ナラティブ引力）が
実装に入り込む。工程を分離することでバイアスの混入を層ごとに管理できる。

## 3. 変則ワークフロー：タスク別

### 公開情報が少ない / 非主流なトピック

```
Grok（主力） → Claude（実装）
ChatGPT の主流ナラティブ引力を避けるため、階層化工程を省略する
```

### Google サービス連携が必要なタスク

```
Gemini（Drive/Docs/Sheets 検索） → Claude（実装）
```

### リアルタイム X データが重要なトピック

```
Grok（X 優先） → Claude（実装）
```

### 長尺 PDF / 大量文書処理

```
Gemini（長尺コンテキスト処理） → ChatGPT（構造化） → Claude（実装）
```

## 4. FAQ

**Q: Claude に最初から全部やらせてはいけないか？**

A: 一人で作業する小タスクなら Claude 単体で問題ない。
問題が起きるのは「一次情報収集 → 構造化 → 実装」を Claude 一人にやらせる場合。
Claude の主流ドリフトが情報収集段階に入ると、後工程全体に伝播する。

**Q: Grok の情報は信頼できるか？**

A: 信頼性ではなく「広さ」に価値がある。Grok の出力を一次ソースとして扱わず、
「この方向に情報があるらしい」という探索結果として扱う。検証は Claude / Gemini で行う。

**Q: このフローは変更可能か？**

A: 当然。各 AI は更新される。このドキュメント自体が v0.1.0 であり、
AI の能力更新ごとに適正配置は変わる。フォーク・PR 歓迎。
