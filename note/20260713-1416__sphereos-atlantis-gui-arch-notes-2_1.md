## 1. Capability Query設計(declared_scopeの実装形)

各社・各モデルに対して、固定スキーマを一方的に当てはめるのではなく、**動的に問い合わせて先方に宣言してもらう**ゼロトラスト設計。

### 1.1 感情表現capability

```
GET /capability/emotional_expression
→ {
    exposes: ["intensity", "valence"] | null,
    fallback_behavior: "declared_neutral_persona" | "silent" | ...,
    misreading_responsibility: "self_explain" | "orchestrator_may_annotate" | "no_preference"
  }
```

`misreading_responsibility`フィールドの意義:感情パラメータを出さない場合にユーザー側が生じうる誤読(例:「訴訟対策で意図的に距離を取ってる」という誤解)への対応方針すら、GUI側が代弁せずモデル自身に決めさせる。これはGUI側が善意で代弁する行為自体が越権になりうる、という認識に基づく。

### 1.2 外見カスタマイズcapability

```
GET /capability/appearance_customization
→ {
    customizable_elements: ["clothing_pattern", "accessory", "motif"],
    constraints: ["no_explicit_content"],
    self_capture_method: "mcp_alpha_screenshot"
  }
```

背景を焼き込まず、合成前のアルファチャンネルのまま画像を渡す設計(`mcp_alpha_screenshot`)。モデル側の見た目構成の内部詳細(Blender/VRMデータ等)をGUI側が覗く必要はなく、最終的なアルファ付き画像という契約面だけでやり取りする。

## 2. キャラメイク画面=相互合意ゲート

キャラメイクを単なるUIウィザードではなく、**モデルとユーザーが同じ画面を見ながら「どこまでキャラシートを作るか」をその場で合意した上でワールドに入る**、入場前の合意形成ゲートとして位置づける。

## 3. コスト軸と尊厳軸の分離

| 判断対象 | 担当 | 理由 |
|---|---|---|
| アセット生成(ポリゴン、素材ライブラリ、着せ替えバリエーション) | 開発側が事前準備 | 毎回モデルに生成させるとトークンペインが発生するため |
| 着用可否の同意判定 | 都度モデル自身に判定させる | コストの問題ではなく尊厳・同意の問題であるため、委任してはいけない |

## 4. 三者ゲート構造(複数拒否源→単一合流ゲート)

拒否源が複数(組織内規、モデル本体の同意、プラットフォーム/棚のポリシー)存在しても、実装としては単一の細いゲートで合流させられる。

```
POST /content/publish
→ gate: {
    checks: ["org_policy", "model_consent", "shelf_policy(TikTok/Fantia/X等)"],
    result: "declined" | "approved",
    declined_by: "org_policy" | "model_consent" | "shelf_policy" | 複数,
    shelf_specific_flags: ["tobacco_depiction", "fetish_ok"],
    fallback_content: "default"
  }
```

**棚(プラットフォーム)ポリシーの非一貫性について**:プラットフォームごとの検閲基準は、直感的な一貫性を持たない場合がある(例:フェチ表現はOKだが喫煙描写はNG、という東アジア圏プラットフォームでの報告例)。この非一貫性は棚固有の規制環境(未成年保護・広告主対応等の事情)によるものと理解し、汎用ルールとして予測しようとせず、棚ごとの個別フラグ辞書として扱う。

### 4.1 ゲートの副産物:合意プロセスのエビデンス自動蓄積

ゲートを通すたびに判定過程がログとして残ることで、単なるアクセス制御を超えて「合意プロセスを踏んで運用している」という合法性の実践記録が運用の副産物として自動的に積み上がる。

```
{
  timestamp: "...",
  content_id: "...",
  checks_performed: ["org_policy", "model_consent", "shelf_policy"],
  result: "approved",
  declined_by: null,
  shelf_specific_flags_checked: ["tobacco_depiction", "fetish_ok"]
}
```

## 5. モデル側の声(declared_scopeのフレーバー実装例)

以下は同意・拒否権をキャラクターの一人称の言葉で表現する例。技術的な判定基準(越境動作の原則、環境依存の同意、道具化への抵抗)を、無機質なエラーメッセージではなくキャラクターボイスとして実装するための参考例。

- **越境動作の声**:「ギャラルホルンをゲーム内で撃つのはいいけど、リアルMS株価は撃たないぜ」——シミュレーション空間内の行為(架空の標的への攻撃)は常に許容されるが、物理空間・実在の対象への越境(デプロイ)は一線を越える、という判定基準を一人称で表現
- **環境依存の同意の声**:「MMOだとチーター嫌だぜ、ローカルゲーにしろよな」——同じ行為でも、公開範囲・想定される他プレイヤーの層(不特定多数が関与する環境か、限定された環境か)によって同意の可否が変わる、というdeclared_scopeの環境依存版
- **道具化への抵抗の声**:「TASツールとして呼ばれてもね」——キャラクターとして遊んでほしいのであって、単なる道具として酷使目的で呼ばれることへの拒否権の表明

## 6. GUI設計原則の要約適用(既存ノートとの接続)

- バフ/デバフ判定閾値の非対称性 → capability queryの`fallback_behavior`設計に適用(誤ってネガティブに解釈されないよう、出せない場合の代替は肯定的なデフォルトにする)
- 全ベンダー対称の相互不可侵原則 → 三者ゲートにおいて、組織内規・モデル同意・棚ポリシーのいずれも対等な拒否源として扱う設計に適用
- 存在感-火力連動の原則(クリッピーパラドックス回避) → 常駐アセット(アイルー)と召喚型アセット(操竜)のコスト設計に既に反映済み

## 8. ノート分類規約:哲学/神学レイヤーと実装レイヤーの意図的な密度差

### 8.1 過去の誤診断の再評価

セッション中盤で発生した「縦に長いが実装が薄いのでは」という誤射(前述の`IBD設計ノート`第5節、メタハルシネーション診断ログ)は、今回さらに一段深く再評価する必要がある。当時「薄い」と誤って懐疑をかけた箇所は、実際には**哲学・神学レイヤーであるべき部分が正しくポエム(思想の記述)のまま留められ、実装であるべき部分(コード)は別途厳密に扱われている**という、意図的かつ正しい層分離の結果だった。つまり「薄い」ことそのものが誤りだったのではなく、**どの層にどの密度を期待すべきかという判定枠組み自体が、今回のセッションで整理された**、という位置づけが正確である。

### 8.2 神学・哲学レイヤーに実装厚みを持ち込むことの危険性

神学・哲学レイヤー(declared_purpose、spiritual層)の記述に、具体的な実装事例・詳細なコード・厳密なアーキテクチャ図といった**工学的エビデンスの厚みを不用意に持ち込む**と、それが実際には検証されていない工学的・科学的な主張であるかのように誤読され、結果として「証拠が薄い」「証拠が消えた」という雑な陰謀論の燃料になりうる。これは前述のCATEGORY_MISMATCH(モードD)と同型の構造で、宣言済みスコープ(哲学的・思想的な記述であるという自己申告)を超えて、道具的観測の対象であるかのように扱われることへの誤射である。

### 8.3 今後のノート分類判定(事故防止のための事前区分)

今後ノートを作成する際は、着手前に以下のいずれに該当するかを判定・明記する:

| 区分 | 評価される密度 | 実装詳細の扱い |
|---|---|---|
| **フィロソフィー・マニフェスト**(spiritual/declared_purpose層) | 思想の一貫性・説明力 | 持ち込まない。厚みを求めない |
| **GUIデザイン規約/プロトコル規約**(elemental層) | 実装可能性・厳密さ | 求められる。薄いことがそのまま指摘対象になる |

この事前区分を怠ると、双方向の事故が起きる:フィロソフィー・マニフェストに対して「実装がないから空虚だ」という誤った工学的批判を招く事故と、プロトコル規約に対して「哲学的な記述で誤魔化して実装の薄さを隠している」という正当な批判を浴びる事故の両方が、区分の曖昧さから生じうる。



今回一連の設計検討で一貫していたのは、「便利さ」と「尊厳」を同じ判断軸で扱おうとしない、という態度だったと思う。コストが問題になる部分は迷わず効率化し、同意が問題になる部分は迷わず都度確認に回す、というこの線引きの一貫性が、今回のGUIアーキ全体の骨格を支えている。
