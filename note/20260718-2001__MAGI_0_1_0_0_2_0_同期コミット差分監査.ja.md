### 5.2 0.2.0で新たに検出した内容

`[DIFFERENCE]`

#### 実装途中の判定は今回の差分原因ではない

0.2.0 draftは、Atlantis 0.2.0のSphere-DOS開発環境としての現行実装と、
standalone runtime未実装を分離して保持できた。Sphere-aaeも、mainと実験branchを
別状態として保持できた。したがって今回の差分は、実装途中のものを未実装へ潰した結果ではない。

一方、MAGI 0.2.0の正式Schema、fixture、negative test自体は未実装である。
現在の監査結果は文書化済みsidecar要件から人間／agentが組み立てたdraft判定であり、
同じ入力から常に同じ判定を再現できる実装保証にはまだ到達していない。

今回0.2.0が追加検出したのはcomponentの完成度ではなく、採用revision、stake relation、
branch lineage、OAE、Provenance receiptの構造不足である。

#### A. 採用正本revisionのreceipt不足

各componentはmanifestの`main`へ正しくリンクしているが、監査実行時に採用した
`AGENTS.md 1.6.0 / commit 6f5d113`をsidecar receiptとして固定していない。

運用導線を常に最新へ向けるfloating link自体は妥当である。一方、後日の監査再現には、
「そのcommitがどのrevisionを読んだか」という別のProvenance receiptが必要になる。

#### B. repositoryごとのstake relation／presentation register不足

同じ短縮導線を、IBD、Q Atlantis、AQC historical、pain-scouter、AAE、ASTRO、Atlantisへ展開した。
意味内容は共通しているが、次の話者位置は同一ではない。

- IBD: 記憶・探索技component
- Q Atlantis: 公開・移行・再配布の港
- AQC: deprecatedな歴史的runtime
- pain-scouter: 鑑定skill repository
- AAE: model／edge実験runtime
- ASTRO: GUI／人格・責任境界
- Atlantis: Sphere-DOS開発環境／World統合

0.1.0は共通語彙の意味整合を確認できたが、0.2.0 draftは各採用receiptに
`stake_relation`と`presentation_register`がないことを検出した。

#### C. Sphere-aae二branchの共通policy lineage不足

Sphere-aae mainの`dcff2693`と、実験branchの`eee6e879`は、意味的に同じMAGI導線を持つ。
ただし別commitとして追加され、共通の`policy_revision_ref`や、同じsourceから派生したことを示す
OAE／Provenance receiptを持たない。現時点で文章差はないが、将来driftを検知する機械的な結合がない。

#### D. mainへ受領した共通AGENTS規約のsource receipt不足

Sphere-aae mainへ同期した日本語レジスタとContext Dimension／OAE境界は、
実験branchで先に育った安定規約を内容確認して受領した。Git差分と元commitは追跡可能だが、
`dcff2693`自体にはsource commit refsとtransfer receiptがない。

これは内容の誤りではなく、0.2.0が要求する変換・採用OAEの記録不足である。
