# リポジトリ構造・内容更新提案

作成日: 2026-07-13

## 1. 背景

既存のノート群、特に [note/20260713-1531__Quantaril Cloud Atlantis世代 — 文書体系マップ(ハブ).md](note/20260713-1531__Quantaril%20Cloud%20Atlantis%E4%B8%96%E4%BB%A3%20%E2%80%95%20%E6%96%87%E6%9B%B8%E4%BD%93%E7%B3%BB%E3%83%9E%E3%83%83%E3%83%97(%E3%83%8F%E3%83%96).md) の内容を読むと、
ZeroRoomLab-manifest では「生のノート」「正規化済みドキュメント」「転送候補」の役割を分ける構造に寄せるべきだという方向性が明確です。

現状のリポジトリ構造は、
- [docs/philosophy](docs/philosophy)
- [docs/theory](docs/theory)
- [docs/projects](docs/projects)
- [note](note)

という骨格がすでにあり、方向性としては妥当です。ただし、以下の点で更新の余地があります。

- ノート群が「入力・保管・棚卸し」の役割を担っている一方、リポジトリ内の「正規化済みドキュメント」の入口が弱い
- 他リポジトリへ転送する内容が、どのレベルで正規化されるべきかの運用ルールが未整理
- ハブ文書と各ストリーム文書の関係を、リポジトリ全体の構造として明文化していない

---

## 2. 追加すべき構造

### 2.1 役割分離

次の3層に分けると整理しやすいです。

1. 収集・保管層: [note](note)
   - 生のノート、ドラフト、非正規化メモ、Drive/Downloads由来の候補を置く
   - 変更履歴と元データを残す

2. 正規化・公開層: [docs](docs)
   - 哲学・理論・プロジェクトの正規化済み文書を置く
   - 代表的な内容はここを更新対象とする

3. 運用・転送層: [docs/operations](docs) もしくは [note/transfer_plan](note/transfer_plan)
   - 転送ポリシー、対象リポジトリ、ステータス表、承認ルールを置く
   - 他リポジトリへ渡す前のキューとしても使う

### 2.2 推奨ディレクトリ案

- [docs/philosophy](docs/philosophy): 哲学・思想・世界観
- [docs/theory](docs/theory): FAM / SphereOS / IBD / Infoton の技術系
- [docs/projects](docs/projects): 実装・実験・プロジェクト管理
- [docs/operations](docs/operations): 運用・ガバナンス・転送計画
- [note](note): 生ノート・ドラフト・参照元の保管
- [note/transfer_plan](note/transfer_plan): 他リポジトリへの転送計画

---

## 3. 追加・更新すべき内容

### 3.1 追加したい文書

- 仕様・運用ガバナンス文書
  - 「ハブ文書とストリーム文書の責務分離」
  - 「ノート→docs→他リポジトリの更新ルール」

- 転送キュー文書
  - どの内容をどのリポジトリへ送るか
  - 各要素のステータス
  - 反映済み / 未反映 / 保留の区別

### 3.2 更新したい既存内容

- [README.md](README.md) と [README.ja.md](README.ja.md)
  - 「このリポジトリは単なるmanifestではなく、ノート・正規化文書・転送キューを持つ知識基盤である」ことを明示
- [docs](docs) 配下の目次・リンク整理
  - 各文書が「原典」か「転送候補」かを明示

---

## 4. 運用方針の提案

### 4.1 原則

- 生のノートは [note](note) に置く
- 他リポジトリに反映する内容は、まず [note](note) で整理し、承認後に正規化して移す
- 1つの内容を複数箇所で二重管理しない
- 変更のあった箇所は、ハブ文書側で参照先だけ更新する

### 4.2 ステータス表

| 状態 | 意味 |
|---|---|
| [DRAFT] | まだ整理中 |
| [REVIEW] | 反映前にレビュー待ち |
| [CANONICAL] | 正規化済みで主文書 |
| [TRANSFER-QUEUED] | 他リポジトリへ送る候補 |
| [TRANSFERRED] | 送信済み |

---

## 5. 優先度

1. [note/transfer_plan](note/transfer_plan) を正式な運用ディレクトリとして整備する
2. [docs](docs) 配下に「運用・ガバナンス」層を追加する
3. 既存の README に「ノート→ドキュメント→転送」の流れを追記する
4. 他リポジトリに送る文書を、段階的にキュー化する

---

## 6. 結論

ZeroRoomLab-manifest は、単なるmanifestではなく、複数リポジトリへ情報を配るための「知識基盤・転送ハブ」として再設計するのが妥当です。特に、ノート群の保管場所と、正規化・転送先を明確に分けることで、今後の並行編集・再配布の混乱を減らせます。
