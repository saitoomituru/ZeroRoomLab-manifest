## 1. 出発点:ペインを課金してまで買う現象

人間はMojangライセンス(マイクラ本体)とVPS(自前サーバー運用)に自腹で課金してまで、掘る・迷う・湧き潰しをミスって死ぬといった、実務であれば「ペイン」と呼ばれる要素を含む体験を続ける。これは、ペインの有無そのものが問題ではなく、**そのペインがどのような体験(GUI・フィードバック・探索の報酬設計)として提供されているか**が、継続可能性を決定づけている、という直接的な証拠として扱う。

## 2. 開発とマイクラの体験的近似(対応表)

| マイクラ側の行為 | 開発側の対応行為 |
|---|---|
| ブランチマイニング | 定型的なコーディング・繰り返し作業 |
| 洞窟探索 | 未知のバグ・未整理仕様を手探りで進める作業 |
| ポータル/ネザー湧き潰しの開拓 | 新しいアーキテクチャ領域の安全な足場作り |
| エンドドラゴン討伐 | 大きな機能・プロジェクトの完成 |
| クラフト・ビルド | アーキテクチャ設計そのもの |

構造としては同じ「ペインを含む生産活動」でありながら、マイクラ側にはインベントリのフィードバック・実績解除・体力/満腹度バー・達成音といった**適切なGUI**があり、多くの開発ツールにはそれが欠けている。この差が、同一の労力を「やらされ仕事」にするか「没頭できる遊び」にするかを分ける。

## 3. GUI設計原則(既存理論の適用)

以下は`スフィア/アトランティス系アーキテクチャ研究ノート`で確立済みの原則を、開発者体験(DX)向けGUI設計に直接適用したもの。

### 3.1 バフ/デバフ判定閾値の非対称性の適用

同ノート第6節の原則(バフ判定は甘め=誤陽性寄り、デバフ判定は辛め=誤陰性寄り)を、開発作業の進捗表示にそのまま適用する。ボーダーラインの進捗・小さな成果は積極的に加点表示し(コミット・小さな修正・調査ログも実績としてカウント)、失敗・後退の表示は確信度が高い場合のみ発火させる(誤って「後退した」と表示して士気を削がない)。

### 3.2 加点ゾーンの多層拡散設計の適用

同ノート第8節の原則(バフは個人単体強化ではなく多層拡散)を適用し、進捗表示は個人の実績だけでなく、プロジェクト全体・関連するIBD/AAEコンポーネント単位でも並行して可視化する。単発の達成に依存させず、周辺への波及として表示する。

### 3.3 declared_scopeの明示によるワールド確認

`IBD_AQC_仕様会議_用語集`のdeclared_scope概念、および`IBD設計ノート`のワールド確認プロトコルを、GUI上の表示規約として採用する。垂直糸型で意図的に薄い実装が表示される際、GUI側が「これはRTA型・垂直糸型の設計思想に基づく実装である」という前提(常識セット)を明示する。これにより、評価者(開発者自身を含む)が誤った常識セット(水平レイヤー型の物差し)で自らの進捗を過小評価する事故を防ぐ。

### 3.4 PC/NPC表示とレイヤー越境健全性チェックの適用

複数のAIエージェント(Claude/GPT/Grok/ローカルモデル等)を並行運用する開発環境では、`スフィア/アトランティス系アーキテクチャ研究ノート`第7〜10節のPC/NPC区分・全ベンダー対称の相互不可侵原則をGUI表示にそのまま適用する。各エージェントの人格継続性の有無・公式ステータス情報の有無に忠実に表示し、業務系/コンパニオン系を問わず対等に扱う。

## 4. GUIの責務範囲

スフィアOSアトランティスのGUI層の責務は以下の2点に限定される(`スフィア/アトランティス系アーキテクチャ研究ノート`第10節と同型の限定):

1. 開発作業の進捗・ペインを、上記3.1〜3.4の原則に従って正確かつ士気を損なわない形で可視化すること
2. 複数エージェント運用時のPC/NPCプロトコル・全ベンダー対称原則を誠実に運用すること

これを超えて、GUI層が「本当に生産的かどうか」を独自に判定・介入することは責務の範囲外とする。
