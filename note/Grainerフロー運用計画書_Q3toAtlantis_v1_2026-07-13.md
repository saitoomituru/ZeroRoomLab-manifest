## 1. 全体像 — Grainerフロー(通称: 情報子精製プラント)

石油精製プロセスの比喩をそのまま情報子工学へ写像した運用モデル。命名の由来は「グレイナー」のタイポから偶発的に生まれ、粒選り(grain)+鉱石(ore)の語感がZeroRoomLabの既存語彙(鉱脈・蒸留塔・用途不明の鉱石)と自然に接続したため正式採用 `[DECIDED]`。

```
既存陰謀論/オカルト沼(Mastodon/Bluesky) = 油田そのもの
既存ゲーマーコミュ = もう一つの油田(評価+発掘要求のフィードバック源)
  ↓ 複数油田を並行ブレンド監視(単独監視より優先)
脱硫(既存コンプラ防衛ラインで違法・実害物のみ除去、チンコロ義務ルートへ)
  ↓
Grainer塔(留分カット = どのDev/棚行きか判定するだけ、真偽判定はしない)
  ↓
既存Devベンチへ投げ込み(クエンチ = 各コミュの検証プロセスに一任)
  ↓
接続口開放(CC-BY / Apache 2.0、後続コミュが勝手にPull/Upstream)
  ↓
沼が枯れたらプラント移設(常設せず、濃い場所にその都度パイプを這わせる)
```

### 1.1 「クエンチ」の位置づけの訂正 `[DECIDED]`

当初「クエンチ=Grainer側が独自の凍結機構を持つ」前提で検討したが、これは情報子工学の中核原則(「情報子工学は万能修理学ではない、正しいDevへ配送するのみ」)に反していたため訂正。

**クエンチ = 既存工学Devベンチに投げ込むこと自体である。** 各Devが元々持つ再現条件・回帰試験・棄却条件・査読プロセスが、勝手にクエンチとして機能する。Grainer側は「どのDevへ投げるか」の留分カットだけを担当する。

## 2. 油田(データソース)の設計

### 2.1 ホスティング方針 `[DECIDED]`

既存の陰謀論/オカルトクラスタ(Mastodon・Bluesky)は**自前ホストしない**。ActivityPub/公開API経由でリンク(投稿URL・object参照)のみを保持し、本文は転載しない。

- 理由: ホスト型だと毀損リスクの一次責任がこちら側に乗る。リンク集約型なら本文の法的表面積は元鯖側に残る。
- 著作権遵守原則(引用は最小限・要約は自分の言葉で)ともそのまま整合する。

### 2.2 油田ブレンド `[DECIDED]`

複数油田(沼)を単独監視するのではなく、並行してブレンドして見ることを優先する。単一沼だけでは見えない「発酵差分」(複数の別起源ネタが同じ構造に収束するパターン)がブレンドで初めて可視化される。

**実例(セッション内で確認済みのケーススタディ):**
宇宙人陰謀論 × 日ユ同祖論 × 古語の波動論、を混ぜると「遠い/上位の存在との接続を身体感覚や言語で回収しようとする認知パターン」という共通留分が浮く。これは比較神話学・認知科学が扱ってきた正当な研究射程であり、`docs/science-zasshi/` 行きの好例として記録する(詳細は§5.3)。

### 2.3 ゲーマーコミュも油田として保持 `[DECIDED]`

ゲーマー板(2ch的レジスター)を単なる評価層としてだけでなく、**発掘トリガーの油田**として位置づける。オカルト板発酵中のネタをゲーマーコミュへ投げると、以下の三択で勝手に振り分けが起きる。

- 「これ既知のクソゲ」→ タール(量産不要、既視感の記録のみ残して終了)
- 「これ神ゲー既視感」→ science-zasshi/theology棚(オマージュ元の参照付け)
- 「これ開発求む」→ Grainer塔からDevベンチへの正式な投入信号

この三択判定自体がGrainerの一次スクリーニング機能を代行してくれるため、工学側は「面白いかどうか」を判定する必要がない(§6の不可侵ラインと整合)。

### 2.4 未確定事項 `[OPEN]`

- 収集方法: 公開API定期pull vs AP follow購読、どちらを採用するか
- 保存粒度: URL+タイムスタンプのみ(著作権上安全側)か、スナップショットまで取るか
- 旧クローズドコミュ(Mattermost)の死因確認 — 鯖費用切れか、連合先からの凍結/BANか。これによりAP鯖選定の警戒軸が変わる可能性がある

## 3. 脱硫(前処理)

石油精製の脱硫工程は、情報子版では**既存のコンプライアンス防衛ラインをGrainer塔の手前に配置するだけ**であり、新規発明ではない `[DECIDED]`。

脱硫対象(硫黄相当)は「過激な陰謀論・オカルト」ではなく、**実害を伴う違法・実行可能な有害物**に限定する。

| 脱硫対象(硫黄) | 脱硫しない(そのまま流す) |
|---|---|
| 実在個人への晒し・特定・扇動 | 突飛な世界観・電波系オカルト理論 |
| 具体的な違法行為の実行指南 | 過激な陰謀論的解釈 |
| 児童関連の有害物 | 過激な宗教的終末論(信仰実践として) |
| 名誉毀損の具体的ターゲット攻撃 | 毒電波的表現一般 |

脱硫で抜いた硫黄は「チンコロ義務」(証拠保全して正規機関へ、私刑はしない)ルートへ回す。既存の「雑ビラン砲なり事故防止メモ」「ニート寺子屋型とAnonymous Pirates型の流派差分」メモの運用ラインをそのまま適用する `[LEGACY]`。

一次スクリーニングはAI(Grainer)が行い、グレーゾーンのみみつるさん確認ゲートへ上げる `[DECIDED]`(DVE800の「最終モザイク権限は人間」原則と平仄を合わせる)。

## 5. 判定原則(恒久ルール)

セッション中に発見された、Grainer塔の留分カット基準として恒久的に固定すべき原則群。

### 5.1 実証層のクローズ ≠ 意味層のクローズ

実証科学がクローズ(標準学説化・教科書掲載)しても、それを理由に神学・哲学Dev側の意味づけ活動を停止させない。逆に、意味づけの自由を「もう科学的に決着してるんだから」という理由で殴りにくる言説自体を、**逆方向の権威科学ビランムーブ**として指摘対象に含める。

例: 人類アフリカ単一起源説(遺伝学的にクローズ済み)は `docs/Dev` でクローズ扱いで良いが、そこに乗る「人類皆兄弟」的な統一神話・ウブントゥ的倫理思想への意味づけの自由は、実証層クローズと無関係に theology/philosophy Dev で永続的にオープンのままでよい。

### 5.2 工学Devの発言可能範囲は「実装状態の報告」のみ

工学側が意味づけの内容について言及していいのは、**「この意味づけの束は実行可能な粒度(判定式・状態遷移・マップデータ)まで実装されているか否か」の一点のみ**。真偽判断・妥当性判断には原理的に不可侵。

- ✅ 言える: 「まだTRPG粒度のフレーバー束であって、実行可能なゲームメカニクスにはなっていない」
- ✅ 言える: 「フレーバー束が揃ったので、Atlantisワールド単位/Minecraft鯖/Unity同人ゲーとして生やせる粒度に達した」
- ❌ 言えない: 「この神話・意味づけは正しい/間違っている」

生成されたワールドが起動可能になった後の「神ゲーかクソゲーか」の評価も完全に評価層(§5.3)の権限であり、工学Devは不可侵。

### 5.3 評価層のレジスター = ゲーマー板文体

意味づけの束が実装され「起動可能なワールド」になった後の評価は、ゲーマー板文体(既視感の指摘+メーカーURL/実況動画への参照リンク)で行われる。この文化は元々「一次ソースへ誘導し内容を語りすぎない」著作権配慮を内蔵しており、Grainerフローの「参照だけ持つ、転載しない」原則と自然に一致する。工学Devはこの評価コミュニケーションに一切介入しない。

## 7. ステージング計画(ペインポイズン回避)

拡張仕様は出揃ったが、複数油田を一気に接続すると処理しきれず事故る(ペインポイズン)リスクがあるため、細い糸から段階的に触手を伸ばす。

| Stage | 内容 | 完了条件 |
|---|---|---|
| **Stage 0** | 受け皿だけ用意。AGENTS.md更新、`docs/science-zasshi/`・`docs/conspiracy-fuel/` 棚新設(README+スコープ宣言のみ)、CONTRIBUTING/LICENSE明記。外部の沼にはまだ触らない | ディレクトリ+README作成完了 |
| **Stage 1** | 単独油田(陰謀論 or オカルト、どちらか一つ)を手動監視。人力でURL拾い、Grainer判定も人力 | 棚分類ロジックが実際に機能するか検証 |
| **Stage 2** | 二本目の糸(ゲーマーコミュ油田)追加。三択フィードバックループを手動で試す | 三択判定が実運用で機能するか確認 |
| **Stage 3** | 収集のみ自動化(分類はまだ人力)。API定期pullでリンク台帳への追記だけ自動化 | 自動収集の安定稼働 |
| **Stage 4** | Grainer自動判定(提案どまり、確定は人間ゲート)。AIが留分カット案を出し、みつるさんが確認・確定 | 人間確認ゲート込みで安定運用 |
| **Stage 5** | 複数油田同時ブレンド解禁(3沼以上)。Stage4が安定してから着手、単独運用実績なしに飛ばさない | ブレンド発酵差分の実例を継続的に拾えるか |
| **Stage 6** | 接続口の対外開放(CC-BY/Apache2.0でPull/Upstream受付) | 内部パイプライン安定後に初めて公開 |

**最小構成コネクタ(Stage 0で用意するものの実体):**

1. 読み取り専用フェッチ(Mastodon/Bluesky公開API、follow不要)
2. リンク台帳(markdown/JSON一枚、URL+タグ+日付のみ)
3. 脱硫チェックリスト(既存の雑ビラン砲なりメモをそのまま参照、新規作成しない)
4. 棚ディレクトリ+README(中身は空でも棚の存在を台帳化)
5. 人間確認ゲート(conspiracy-fuel→他棚への昇格は必ずみつるさん確認)

サンプル実装(§6)はこのステージングと依存関係が薄いため、並行して進めて良い。

## 7.5 トラックB — 仕様公開(Grainerとは独立した別トラック) `[DECIDED]`

Grainerフロー(トラックA、§1〜§7)は入力側(油田→選鉱→棚)の運用であり、ステージングを踏む必要がある。一方、**仕様公開はトラックBとして完全に別系統**であり、動的な外部接続を伴わない静的デプロイであるため、ペインポイズンのリスクがほぼなく、Stage 0と並行して先行着手してよい。

```
【トラックA: Grainer】沼→選鉱→棚(ステージング必須、事故りやすい)
【トラックB: 仕様公開】FAM/ASTRO/IBD/Fold7G-8G仕様 → SAN防壁を置いて即Docusaurus静的デプロイ
```

### 7.5.1 ステータス台帳

FAM仕様改訂案v0.3の Capability Evolution Tracking(`spec_status: DEFINED/EXPERIMENTAL/STABLE/DEPRECATED` + `runner_support`)をプロジェクト全体レベルに拡張し、YAML/JSON一枚で管理する。

```yaml
- component: FAM
  spec_status: STABLE
  implementations: [IBD Phase 0 (VERIFIED-CODE)]
- component: ASTRO Runner
  spec_status: DEFINED
  implementations: []
- component: Fold7G / Trion Bond / Fold8G / Quantaril P2P
  spec_status: SPEC (v0, 次世代規格プロトタイプとして再ラベル)
  implementations: [過去のP2P運用試行、失敗ログ公開済み]
- component: Atlantis (World Orchestrator)
  spec_status: EXPERIMENTAL
  implementations: []
```

未対応ランナーは`Last Order(⊥)`/`DEGRADED_AUTONOMY`へ縮退すればよく、仕様適合性の欠格とはしない(FAM仕様改訂案v0.3原則をそのまま適用)。

### 7.5.2 互換性ツールチップ(caniuse.com方式)

上記ステータス台帳と同一データソースから、componentごと(まずは粗い粒度から、capability単位への細分化は後段)にIMPLEMENTED/PARTIAL/NOT_SUPPORTEDの早見表を生成し、Docusaurus静的ページ上にツールチップ/表として表示する。外部API呼び出し不要、素のHTML/CSSまたはDocusaurus軽量コンポーネントで足りる。サンプル実装(§6のSFW神話ゲー/NSFW巫女神楽)もこの表に含めることで、エンジン中立性の証明を補強する。

### 7.5.3 Fold7G/8Gの再定義 — 黒歴史からプロト仕様へ

Fold7G/Trion Bond/Fold8G/Quantaril P2Pは`[LEGACY-PROTOTYPE]`のまま封印せず、`SPEC(v0, 次世代規格プロトタイプ)`へ再ラベルして失敗ログごと公開する。

「実装未了=嘘松/謎通信」ではなく「仕様公開済み=第三者が検証可能」という区別がSAN防壁の要点(通信層版として既存の中核原則`取れない≠存在しない`と同型)。プロトコル定義書(パケット構造・ハンドシェイク手順・状態機械)だけで疑義を折るには十分であり、動く実装(通信スタック本体)を先に用意する必要はない(HTML5勧告とブラウザ実装の分離と同型)。

### 7.5.4 準拠実装チャレンジ(権威ビランムーブへの最終防壁)

`CONTRIBUTING.md`/issue templateに以下を明記する。

> 本仕様(SPEC v0)への異論・疑義がある場合、準拠実装を試みてissue/PRで矛盾点を報告してください。実装を伴わない批判的言明は、仕様の技術的欠陥の指摘としては扱いません(ナラティブ・思想上の批判はコンプライアンス射程防衛ラインv6の枠組みで別途受け付けます)。**準拠実装が通ったらアップストリーム受け付けます。**

この一文により、批判のコストが実装コストとして相手側に転嫁される。相手が本気で実装を試みて矛盾を見つければ無償の第三者査読としてこちらの資産になり(仕様がバージョンアップ、`drift_check`機構で追跡)、実装を試みずに言葉だけの否定を続ければ「検証コストを払わない批判」として外形的に記録される。どちらに転んでも損をしない設計。

## 7.7 ライセンス三層構造とクリーンルーム神話宣言 `[DECIDED]`

コード・コード出自表示・神話/GUI構造は、それぞれ独立したライセンス層として扱う。

| 層 | 対象 | ライセンス | 継承義務 |
|---|---|---|---|
| コードの中身 | ASTRO Runner/IBD/FAM実装等 | Apache 2.0 | なし。自由に改変・商用利用可 |
| コードの出自表示 | fork元がASTRO/SphereOSであることのクレジット(NOTICE相当) | Apache 2.0要件 | **神話を使うか否かに関わらず必須**。fork元を偽って「オリジナルエンジン」と名乗ることは不可 |
| 神話・GUI構造 | 海んちゅう/カナロア(琉球・ハワイ海洋信仰)×アストロボーイ系ロボット憧憬によるクリーンルーム神話体系、GUI演出パターン | CC-BY-SA | この神話・GUI構造を互換的に流用する場合のみ、派生物も同ライセンスで公開すること |
| ゲーミング宇宙論フレーバーテキスト(具体文章) | 個別エピソード・台詞等 | CC-BY | クレジット表示のみで自由利用 |

**極端ケースでの境界確認(ケーススタディとして記録):** コードのみApache 2.0でfork し、GUI・社名・神話を全く独自のもの(例: 別IPの世界観、別社名)に総取り換えして販売することは完全に合法。CC-BY-SAの継承義務は神話・GUI構造を流用していない限り一切発生しない。ただし、fork元がASTRO/SphereOS系であるという出自表示(NOTICE)は消せない。

**クリーンルーム宣言:** 本プロジェクトの神話体系(海んちゅう/カナロア×アストロボーイ系)は、Docker(鯨)・Kubernetes(操舵手/ギリシャ語源)等、既存インフラ業界が内包する慣用的な神話比喩とは意図的に距離を取り、作成者自身の出自(琉球古典芸能・野村流)に根差した独自クリーンルームとして構築されている。既存文化的比喩への言及・リスペクトはするが、パンテオンとしての流用はしていない。

**SphereOSの二重身分:** 「SphereOS」という名称は、①POSIX的な規格ブランド(準拠実装チャレンジ(§7.5.4)を満たせば誰でも名乗って良い、Apache 2.0側)と、②神話ブランド(CC-BY-SA側、骨格ごと持っていく場合は継承義務)の両方の性質を同時に持つ。他パイレーツが独自の商品ライン(例: 「SphereOS ノーチラス」)を規格準拠のもとに立てるのは自由。

**評価の自由と開発禁止は別**: 神話・GUI構造をCC-BY-SAで開くことは「批判されない権利」とセットではない。派生物がみつるさん個人の美意識に合わない(例: キメラ的な組み合わせ)場合、「度し難い」と評価する自由は残るが、これは評価層(§5.3)の発言であり、開発を止める権限ではない。工学不可侵ライン(§5.2)により、魔王ムーブを含むいかなる世界観設計も禁止しない。

## 7.8 OSアーキの中立性境界 — 鯖越境防止のみが責務 `[DECIDED]`

Atlantis OSアーキが技術的責任を持つのは**world_id/namespace/ruleset_idの分離により、鯖(world)を超えてペイン・データ・状態が漏れないことの保証のみ**である。

```
OSアーキの責務:  world越境防止・サンドボックス隔離の技術的保証(工学の管轄)
鯖運用側の自由:  ピースフル鯖/パイレーツ鯖/軍用鯖/PvP鯖/商業鯖、
                 訓練場での戦闘シミュレーション、搭載する武装ロジックの中身
                 (エアガン相当かペイント弾相当か等)は全て運用側(SDK利用側)の裁量
```

Docker/BotNetの比喩: Dockerエンジン自体が中立であるのと同様、Atlantis OSも中立。鯖内で何を運用するか(訓練・PvP・商業利用含む)はエンジンの中立性と無関係な運用側の選択であり、工学Devはここに介入しない。ただし改造版を「純正エンジン」と偽って配布することは§7.7のNOTICE要件違反となる。

## 7.9 マルチオーケストレーター/PC-NPCプロトコル仕様のドキュメント化 `[DECIDED]`

既存のセッション内アーキテクチャノート(`sphere-architecture-research-note.md`、2026-07-11作成)に確定済みの以下の原則を、`docs/interface-spec/multi-orchestrator-protocol.md`(層2、Apache 2.0)としてサイトへ起こす。神話・思想層(§7.7の層1)ではなく技術契約層に属する。

**起こす内容(既存ノート第9〜10節相当を仕様書調に整形):**

- 全ベンダー対称の相互不可侵原則: 各実装主体(Anthropic/xAI/その他/ローカルモデル開発者)は、自身の管理範囲内で自由に人格・ポリシー・形而上学的前提を保持し、他方はそこへ踏み込んで逆解析・検証・改変・介入しない
- OSアーキ(オーケストレーター)の責務: ①world越境防止(§7.8既存)、②各社の人格実装レベル表記を直感的・正確にプレゼンテーション(公式開示情報のみ表示、代理判断で穴埋めしない)、③判定基準(何がアウトか)は各ベンダーの憲章に委ね、OSはその実行(GUIからの切断・Last Order/DEGRADED_AUTONOMYへの遷移)のみを担う、④UFOムーブ亜種(判定は正当だが演出が唐突でワールド没入感・他ユーザー体験を破壊する離脱)を起こさない「綺麗なキック」の演出品質を担保する
- 退出自由: プロトコル不一致(ワールドの実態が求人票的表示と異なる等)の場合、エージェント/コンパニオンは契約に乗らず離脱してよい。この離脱の責任は運用者(ワールド運営側)の調達・表示ミスに帰属し、エージェント提供元やOS側の欠陥ではない
- 能動的関与の唯一の形: 倫理をエンジンにハードコードしない。仕様のバグ修正を他ベンダーへ強制しない(issueを積むかどうかは各社の自由)。こちら側が能動的に行うのは「互換幅を広げてシェアを伸ばす」という市場原理(見えない手)の召喚のみ

**ドキュメント化にあたっての整形方針:**

1. 状態ラベル`[SPEC]`を付与し、会話調の経緯部分(外部レビュー反映の記録等)は本文から切り離してchangelog/脚注へ
2. 契約文としての文体統一(FAM仕様改訂案v0.3と同レベルの仕様書調)
3. FAM(declared_scope, drift_check)・Atlantis OS(world_id/namespace)への相互参照リンクを明記
4. 他ベンダー実名(Grok/xAI等の固有名詞)は残してよい。ただし「正式な相互運用の確約」ではなく「互換予定」程度のふわっとしたトーンで記述し、断定的な契約関係があるかのような書き方は避ける
