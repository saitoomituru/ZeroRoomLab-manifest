## 0. このノートの読み方

この文書は完成済み仕様ではない。今回のブレストで見えた責務、接続方式、未解決事項を、後から実装・分割・反証できる粒度へ落とした作業ノートである。

本文では次の状態ラベルを使う。

- `[PROVISIONAL]`: 現時点で採用方向だが、実装・負荷試験・ライセンス確認前
- `[OPEN]`: まだ選定・検証が必要
- `[BOUNDARY]`: 他コンポーネントへ押し込まない責務境界
- `[DEFERRED]`: Atlantis 0.1.0以降へ送れる要素
- `[POEM]`: 工学仕様へ平板化せず、Layer Bの目的・意味として保存する部分

本ノートの目的は「全部を一つのDBへ入れること」ではない。

**異なる保存系・探索系・業務系を、論理ID、由来、責務、再投影可能性によって一つの作業世界へ接続すること**が目的である。

## 1. 現時点の結論

`[PROVISIONAL]`

IBDは、単なる記憶ストア、単一ベクトルDB、単一グラフDB、単一RDB、あるいはチャット履歴保存器ではない。

IBDは次をまとめる、**メタファー中立な汎用NoSQLメタドライバー／ポリグロット永続化SDK**として再鍛造する。

```text
Application Model / FAM / Skill / Evidence / Artifact
                         ↓
       IBD logical identity + schema + provenance
                         ↓
      query / watch / relate / compose / migrate
                         ↓
  Neo4j | MariaDB | FileMaker | Firebase | External RDB
                         ↓
       graph projection / vector index / evidence reference
```

ここでいう「NoSQL」は、特定製品カテゴリーとしての狭いNoSQLではない。

**アプリケーションが物理DBの表・カラム・レイアウト・接続先へ直接拘束されず、複数のデータモデルを一つの論理モデルとして利用できる抽象機構**を指す。

Firebase的なアプリケーション向け抽象と同系統だが、Firebase API互換やワイヤー互換を意味しない。IBDではさらに、FAM探索枝、情報子クラスター、グラフ再結合、定量エビデンス、外部業務DB参照、スキーマ移行、再投影、由来追跡を第一級に扱う。

## 2. 掘った坑道を、次の猫へ渡す

`[POEM]`

> 掘った坑道は、答えが出なかったから死ぬのではない。  
> 入口を忘れたときに死ぬ。
>
> 昔の職人が残したのは、完成品だけではなかった。  
> 鑿の角度、木の鳴り方、火を止めた匂い、失敗した継ぎ手、  
> 「ここから先は雨の日に触るな」という短い言葉も、技の一部だった。
>
> 情報子クラスターは、その再現可能なところまで切り出した叡智である。  
> スピリットは、その叡智が仕事場へ立ち上がったときに見える職人魂である。
>
> Atlantisは魂を所有しない。  
> 忘れられた技へ入口を戻し、必要な道具を渡し、  
> いまの仕事に合う形へ組み直せる作業場をつくる。
>
> 猫は毎回ゼロから賢くならなくてよい。  
> 昔掘った枝を拾い、違う枝と継ぎ、  
> それでも足りない場所だけを、今日の爪で掘ればよい。

このポエムは装飾ではない。IBDが保持すべき目的関数を示している。

- 答えだけでなく、探索技を保存する
- 成功枝だけでなく、凍結枝・失敗枝・反証枝も保存する
- 過去の枝を検索可能にする
- 別の問題へ移植できる単位へ分解する
- 元の文脈、証拠、停止条件を失わずに再結合する
- GUI上では、技の束を人間が理解できる「スピリット」として扱う

## 3. 用語と二重記述

### 3.1 情報子クラスター（Infoton Cluster）

`[PROVISIONAL]`

再現可能なところまで切り出された、探索技・判断技・作業技・証拠条件・停止条件のモジュール単位。

単なる文章断片や埋め込みベクトルではない。

```text
Infoton Cluster
├─ FAM branch / exploration technique
├─ input pattern
├─ output contract
├─ graph ports
├─ required evidence
├─ required tools / MCP capability
├─ preconditions
├─ stop conditions
├─ failure modes
├─ permission scope
├─ provenance
└─ version / compatibility
```

### 3.2 FAMモジュール

情報子クラスターのうち、FAMの探索・分岐・検証・停止へ接続できる実行可能または解釈可能なモジュール。

FAM全体を巨大な固定グラフとして保存するのではなく、再結合可能な枝として保存する。

### 3.3 合成FAM（Composite FAM）

クエリ、利用可能なツール、権限、証拠源、対象階層に応じて、複数の情報子クラスターを選択・接続して生成される一時的または再利用可能なFAM。

```text
Composite FAM
= selected Infoton Clusters
+ assembly graph
+ evidence bindings
+ tool bindings
+ permission bindings
+ unresolved slots
+ stop conditions
+ provenance
```

### 3.4 スピリット（Spirit）

`[POEM]` `[BOUNDARY]`

GUI上で、情報子クラスターの束を「霊的な職人魂」「この職人が何を知り、何をでき、どこで手を止めるか」として扱う表現。

スピリットは単なるキャラクタースキンではない。一方で、IBDの物理ストレージ構造そのものでもない。

```text
内部実装:
  情報子クラスター + FAM合成 + Skill + Tool + Evidence + Boundary

GUI表現:
  スピリット / 職人魂 / 呼び出せる叡智
```

スピリットの実在、魂、意識、自我の自然科学的真偽をIBDが判定してはならない。IBDが扱うのは、再現可能な構造、由来、接続、実行条件である。

### 3.5 Service Angel Role

Atlantis上で、スピリットまたは作業エージェントへ割り当てる職務・能力・責任境界。

```text
Service Angel Role
= capability manifest
+ allowed MCP tools
+ evidence requirements
+ budget / quota boundary
+ human approval gates
+ prohibited actions
```

### 3.6 CUI SaaS猫

Codex CLI、Gemini CLI、Claude系CLI、ローカルCLIエージェント等、Atlantisが検出・選択してService Angel Roleへ割り当てる外部またはローカル作業主体の総称。

IBDは猫を指揮するオーケストレーターではない。IBDは、猫が使う情報子クラスター、合成FAM、証拠参照、作業資料、成果物由来を供給・保存する。

## 4. FAM探索技をモジュール化し、クエリごとに再結合する

### 4.1 基本フロー

`[PROVISIONAL]`

```text
Query / Quest
  ↓
目的・対象・制約・必要証拠・許可範囲を解釈
  ↓
検索対象のFAM階層を決定
  ↓
各階層で情報子クラスターをベクトル検索
  ↓
Neo4jで接続可能性・依存・禁止条件・過去失敗を確認
  ↓
証拠源、Skill、MCP Tool、Service Angel Roleを束縛
  ↓
不足ポートを未解決スロットとして明示
  ↓
Composite FAMを生成
  ↓
実行後、RecipeとRun Traceを分離して保存
```

### 4.2 「特定階層だけをベクトルで探す」

全情報子クラスターを一つの巨大な類似度空間へ投げ込むと、意味の近さと役割の互換性が混線する。

そこでベクトル検索は、FAM上の役割・階層・ポート型で候補集合を絞ってから行う。

例:

```text
Observation layer       観測・入力正規化
Recall layer            過去枝・類似技の発見
Relation layer          関係展開・グラフ探索
Hypothesis layer        仮説枝生成
Quantitative layer      定量エビデンス取得
Refutation layer        反証・偽陽性・偽陰性検査
Boundary layer          権限・法域・停止条件
Delivery layer          成果物・検品・人間承認
```

新しいクエリに対して、各層から必要なモジュールを拾い、Neo4j上の接続関係で組み立てる。

### 4.3 Neo4jの役割

`[PROVISIONAL]`

Neo4jは、FAMの探索地形と、情報子クラスターの再結合可能性を保持する。

- FAM枝、分岐、統合、停止、反証、由来のグラフ
- 情報子クラスターのノード
- 接続ポート、入力型、出力型、依存関係
- 過去のComposite FAM Recipe
- 実行時Run Trace
- Skill、Tool、Evidence、Artifactとの関係
- 探索技、問い、分岐理由、失敗条件等のベクトル表現
- ベクトル検索で発見した候補から、グラフを展開・型検査する経路

Neo4jにベクトル索引機能があることを前提に、初期構成では独立Vector DBを必須にしない方向を検討する。

ただし、利用可能なエディション、索引対象、関係への適用範囲、ラズパイ上のJVM・メモリ負荷、量子化、再索引時間は実装時に確認する。独立Vector DBアダプターは、将来のスケールアウトまたは互換境界として残してよい。

### 4.4 RecipeとRun Traceを分ける

```text
Recipe
= どの情報子クラスターを、どの順序・条件で組み合わせたか

Run Trace
= 実際の実行で、どの枝を辿り、何を観測し、どこで止まったか
```

成功Recipeだけを保存してはならない。

- 証拠不足で凍結した合成
- 間違った類似枝を拾った合成
- 外部DBが停止して不成立だった合成
- モデル更新で壊れた合成
- 人間が却下した合成

も、次回の反証・再探索・資源到着時の再活性化へ利用できる。

## 5. IBDのデータ面

### 5.1 全体像

```text
Atlantis / ASTRO / Application / Agent
                    ↓
                 IBD API
                    ↓
      Logical ID / Schema / Resolver / Provenance
                    ↓
 ┌──────────────────┼────────────────────┐
 ↓                  ↓                    ↓
Neo4j            MariaDB             Connector Layer
Graph + Vector   Data + Meta         External systems
 ↓                  ↓                    ↓
FAM / relations  Evidence / Skill    Oracle / FM / Firebase
clusters / trace mapping / jobs      ODBC / JDBC / hooks
```

### 5.2 Neo4j: 探索地形・関係・ベクトル

Neo4j側では、「何がどこにあるか」だけでなく、「何と何がどう繋がり、どの探索技がどの条件で再利用できるか」を持つ。

Neo4j上の主キーは、MariaDB等の物理主キーへ直接固定しない。

```text
(:Infoton {ibd_id: "ibd:cluster:..."})
(:Evidence {ibd_id: "ibd:evidence:..."})
(:Skill {ibd_id: "ibd:skill:..."})
```

物理レコードの差し替えはMariaDBのメタ層で吸収し、Neo4jは安定した `ibd_id` を参照する。

### 5.3 MariaDB: 定量データとメタDB

MariaDBは二つの役割を持つ。

#### A. Atlantis管理下の定量・作業データ

- 測定値
- ジョブ状態
- 予算・クォータ記録
- 検証結果
- 作業票
- Skill.md
- RDB的な資料
- 作業資料
- 原文Markdown
- 構造化済みSkill定義
- 成果物メタデータ

#### B. IBD全体のメタDB

- 論理IDと物理レコードの対応
- connector registry
- schema registry
- schema version
- migration history
- projection state
- reindex state
- external record mapping
- Neo4j投影状態
- embedding生成条件
- source-of-record宣言
- webhook / polling cursor
- バッチ実行履歴
- 冪等性キー
- ロールバック地点

MariaDBは、外部DBの全データを複製することを必須にしない。外部レコードへの参照、観測時点、クエリ条件、レコードハッシュ、権限範囲をメタ情報として持てる。

### 5.4 原文と構造化データを両方持つ

Skill.mdや作業資料は、構造化の過程で原文を失ってはならない。

```text
Source Document
├─ raw_text / raw_markdown
├─ content_hash
├─ source_uri / repository path
├─ version
├─ author / generator
├─ observed_at
├─ parser_version
└─ structured projections
```

構造化投影の例:

```text
Skill
├─ Procedure
├─ Constraint
├─ ToolRequirement
├─ InputSchema
├─ OutputSchema
├─ EvidenceRequirement
├─ StopCondition
└─ FailureMode
```

原文、構造化表現、Neo4j関係、ベクトル索引は別の投影として扱い、再生成可能性を残す。

添付ファイルや巨大バイナリは、将来オブジェクトストレージへ逃がし、MariaDBには論理ID・ハッシュ・参照を置く方式も許容する。

## 6. 外部システムを同じ抽象へ接続する

### 6.1 Connector Capability Manifest

IBDは、すべての接続先を「SQLと同じもの」として偽装しない。

各コネクターは能力表を持つ。

```yaml
connector_id: example.production
connector_type: filemaker_data_api
capabilities:
  read: true
  query: true
  create: false
  update: false
  delete: false
  watch: via_hook
  procedure: via_script
  transaction: limited
  schema_introspection: partial
  attachment_read: true
source_of_record: external
secret_ref: secret://connector/example.production
```

IBDからは共通操作へ畳むが、非対応能力は明示的に返す。

```text
get()
find()
put_draft()
watch()
relate()
resolve_evidence()
bind_external_record()
execute_procedure()
```

### 6.2 ODBC / JDBC / 機関業務システム

Atlantis Server Advancedを見越し、次の外部参照を抽象化する。

- Oracle
- PostgreSQL
- SQL Server
- MariaDB / MySQL系
- DWH
- ERP
- 建設・労務・会計等の機関業務システム
- 計測・製造ログDB
- ODBC / JDBC接続先

原則として、外部機関DBはその機関側のsource of recordである。

IBDは必要な定量データを参照し、次を記録する。

```text
EvidenceReference
├─ ibd_id
├─ connector_id
├─ source_object
├─ query_template_id
├─ parameter_hash
├─ returned_record_hash
├─ observed_at
├─ authority_scope
├─ access_scope
└─ reproducibility_hint
```

### 6.3 FileMaker Data API

FileMakerは単なる移行元ではなく、現役の業務ノードとして扱う。

IBDコネクターでは、FileMaker固有の次の境界を失わない。

- ファイル
- レイアウト
- テーブルオカレンス
- レコードID / UUID主キー
- ポータル
- 計算フィールド
- コンテナ
- スクリプト
- アクセス権セット

FileMaker Data APIは、レコードの取得・検索・作成・更新・削除やスクリプト接続のアダプター候補とする。

FileMaker側の業務ロジックを無視して、外から直接テーブルを書き換える設計へ自動変換してはならない。

### 6.4 Claris Connect Hook

Claris Connectは保存本体ではなく、イベント橋として扱う。

```text
FileMaker / SaaS event
  ↓
Claris Connect flow / webhook
  ↓
IBD event intake
  ↓
MariaDB event log / outbox
  ↓
Neo4j projection update
  ↓
必要箇所だけ再embedding / reindex
  ↓
Atlantis Quest発火
```

Hook欠落、停止、重複を前提に、修正タイムスタンプ、カーソル、定期差分、ハッシュ監査等の補助経路を持つ。

### 6.5 Firebase等

Firebase等のmanaged document/event backendも、能力の異なるコネクターとして扱える。

```text
Firebase
= managed document / auth / event backend

FileMaker
= layout / script-aware business backend

MariaDB
= quantitative data + IBD metadata backend

Neo4j
= graph + vector exploration backend

External institutional RDB
= authoritative quantitative/business backend
```

ここでの共通性は、Application Modelを物理保存先から切り離す抽象機構にある。API互換性を自動的に主張しない。

## 7. RDB構造変更の泥沼をIBDで抽象化する

### 7.1 問題

MariaDB等のRDBでは、運用後にカラム、制約、主キー、正規化方針が変わると、単純な `ALTER TABLE` だけでは次の泥沼が発生する。

- 長時間ロック
- 旧アプリとの互換崩壊
- ETL失敗
- 主キー変更
- 外部参照切れ
- Neo4jノードとの対応切れ
- embedding再生成
- vector index再構築
- rollback困難
- 古いデータと新しいデータの混在

### 7.2 Atlantis管理DBの暫定方針

`[PROVISIONAL]`

大きな構造変更では、既存DBをその場で無理に変形するだけでなく、**新しいDBまたは新しいschema namespaceを生成し、流し込み・検証・差し替えをバッチ化する**。

```text
Schema Change Request
  ↓
新schema / 新DBを生成
  ↓
旧DBからETL / transform
  ↓
件数・ハッシュ・制約・サンプル照合
  ↓
Logical ID ⇔ Physical Key対応表を生成
  ↓
Neo4jへの物理参照を再束縛
  ↓
関係投影を再構築
  ↓
embeddingを再生成
  ↓
vector indexを再構築
  ↓
read endpointを切替
  ↓
限定write / canary
  ↓
write endpointを切替
  ↓
旧DBをread-only退避
  ↓
問題時は旧DBへrollback
```

### 7.3 論理IDを物理主キーから分離する

```text
IBD Logical ID
    ↓
Physical Binding
    ├─ connector_id
    ├─ database_id
    ├─ schema_name
    ├─ table_or_layout
    ├─ physical_primary_key
    ├─ record_version
    ├─ valid_from
    └─ valid_to
```

例:

```text
ibd:measurement:abc123
→ mariadb_v3.measurements.id = 4412
```

移行後:

```text
ibd:measurement:abc123
→ mariadb_v4.measurement_records.uuid = "..."
```

Neo4j側は `ibd:measurement:abc123` を見続ける。物理主キーの変更はMariaDBメタDB内のbinding差し替えで吸収する。

### 7.4 source of recordと派生投影

```text
RDB / external system record
= authoritative or declared source record

Neo4j node / relationship
= graph projection

embedding
= semantic projection

vector index
= retrieval projection
```

グラフ、embedding、vector indexは再生成可能な派生物として扱う。

ただし、FAMの人間編集、グラフ固有の注釈、分岐理由等がNeo4j側で一次情報になる場合もあるため、各モデルごとにsource of recordを宣言する。全モデルへ「RDBが常に唯一の正本」を強制しない。

### 7.5 結果整合とOutbox

三つ以上の保存先へ分散トランザクションを強制すると、ラズパイ初号機でもServer Advancedでも運用が硬直する。

初期方針候補:

```text
MariaDB transaction / external observation
  ↓
outbox event
  ↓
Neo4j projection
  ↓
embedding job
  ↓
vector reindex
```

- 各ジョブは冪等にする
- 途中失敗時は再実行できる
- projection lagをメタDBへ記録する
- 古い索引であることを問い合わせ側へ返せる

## 18. Season 0転送後の責務追補

状態: `[TRANSFERRED]` `[PROVISIONAL]`

本ノート作成後の責務検討により、IBDの中心を「汎用NoSQLメタドライバー／ポリグロット永続化SDK」とだけ呼ぶ説明は、物理DB統合へ寄りすぎることが分かった。Season 0では次を中心契約とする。

```text
ψ  = 状況
∇φ = 手法・探索方向
λ  = 目的・到達条件
Q  = 制約・探索範囲・証拠条件
```

IBDはQuery FAMを受け取り、既知の探索技を情報子クラスターとして検索・接続し、最終回答ではなくComposite FAMという探索技のまま返す。

### 18.1 分類と保存先

- 分類語彙、意味境界、閾値、保存先、unknown、評価規約はλを所有する上位システムがClassification Registryとして定義する
- IBDはRegistryの範囲内でFAMLogの枝をベクトル分類する
- Schema Bundleごとに論理的なIBD Databaseを一つ持つ
- `Vendor / System / Assistant / User`、`Vision / TEC / Financial`、`Cloud Chakra / Spirit / Astral / Elemental`、哲学、神学、自我を機能的に等価な第一級Registryとして扱える
- 社会的に一般的な分類を正規形とし、他の意味体系を自由文タグへ矮小化しない

### 18.2 バインダーとしての中立性

`[POEM]` `[BOUNDARY]`

一般的な「中立」は、企業、国家、制度、AI vendorがホワイトと定義した白色顔料である場合がある。IBDの中立性は白色顔料ではない。

> IBDは白を決めない。上位システムが提供した分類機定義の顔料を、その色と由来を失わず結ぶバインダーである。

カレー色、スカイブルー、赤を`white`と定義する自由を妨げない。一方、異なるRegistryの`white`を無断で同一視しない。善悪、美醜、リスク、真偽の評価は上位システムまたはユーザーに属し、IBDは評価主体と評価基準をProvenanceとして保持する。

### 18.3 隔離と非破壊混色

- 混色命令がない限り、Database間の近いベクトルや同名classを候補集合へ入れない
- 上位Qが複数DatabaseとMapping FAMを明示した場合だけ探索技を結合する
- Composite FAMはsource clusterを変更しない派生Recipeである
- 混色結果を永続化する場合は、新しいSchema Bundle／IBD Databaseとして明示する
- 混色結果が美しいか、度し難いかはIBDが判断しない

### 18.4 Last OrderとEvidence鮮度

打ち切り探索では、最後に完了した枝、停止理由、次に必要なEvidence、再開条件をLast Orderとして保存する。RDB等を参照した場合はquery fingerprint、parameter hash、schema fingerprint、result hash、観測時点、依存FAM branchを保持する。後日ソースが伸びた場合は、過去記録を上書きせず、影響を受ける枝と再探索候補を返す。

### 18.5 自我対応と実行形態

自我対応はoptional extensionではなくIBD Driverの必須要件とする。Subject、Actor、AgentInstance、Runtime、SelfModel、ContinuityClaim、BodyBindingを平板化せず扱い、自我の実在や連続性はIBDが裁定しない。

IBDはEmbedded Library、Native Service、Containerized Serviceのいずれでも動作可能とし、Dockerを正体または必須条件にしない。論理IBD Databaseとprocess、container、物理DB serverを同一視しない。

2026-07-18にIBDリポジトリへSeason 0文書、Schema草案、人工Query FAM fixtureとして転送した。

### 18.7 存在論の担保と上位fact scope

`[BOUNDARY]`

IBD Coreは魂、霊、神、自我の存在論自体を担保・否定しない。一方、上位システムが存在を確定した書き込みを、IBD独自の科学・企業・vendor基準で拒否または弱化しない。

MMO上位システムがWorld内の神Entityを実在と確定した場合、IBDはそのWorldとfact scopeで`confirmed`として保持する。神学者・哲学者が自身のRegistryで存在論を提示した場合も、自然科学で未証明という理由で自動降格しない。

```text
existence status
fact scope
World / Context
declared or decided by
Registry / Evidence
```

を組として保存し、Query FAMのQが指定した上位分類機の定規どおり返す。これは神の存在をIBDが普遍的事実として保証することでも、虚構として否定することでもない。異なるWorld・学問・制度のfact scopeを混ぜず、必要なMapping FAMがある場合だけ比較・結合する。

### 18.8 実行環境時計と上位時系列

`[BOUNDARY]` `[TRANSFERRED]`

IBDが持つメタ時計と、上位Systemが管理するデータ時系列を混ぜない。

```text
IBD meta clock
  created / modified / imported / exported / indexed / migrated

upstream timeline
  source created / source modified / occurred / observed
  scheduled / valid period / business period
  game day / turn / tick / narrative position
```

import／exportした当日の時刻で、過去に作られた原資料の時刻や未来予定を上書きしない。game worldの暦やtickを、Mapping FAMなしにUTCへ換算しない。両方を返す場合も、IBDメタと上位時系列であることを明示する。

エッジ環境ではGPS、5G、NTP等を取得する前に、`1970-01-01`だけでなく、マザーボード製造日、vendor出荷日、firmware build日等のもっともらしい初期値が現れ得る。値の見た目では校正済みと判断せず、NTP server名、GPS最終受信、carrier network time、標準電波／FM、RTC module、校正Evidence、不確かさ、boot ID、monotonic clock、sequenceを保持する。同期後はanchor／correction eventを追加し、同期前eventを破壊的に書き換えない。

時計信憑性をどの粒度まで採用するかは上位SystemがQで決める。RTCで十分な用途も、NTP校正後5分以内でなければ不足する用途もある。`calibrated: false`はwall clockの立証範囲を示すmetadataであり、record本文全体が偽であることを意味しない。ただし校正Evidenceがないのに`calibrated: true`とは提示しない。

### 18.9 Schemaへの忠実性とfact非裁定

IBDはSchemaや取得情報へ嘘をつかないが、何が普遍的factかは判断しない。

- MMOの魔王はWorld stateを定規に実在とできる
- 攻略Wikiはquest配信・掲載を定規に実在とできる
- 神学者・哲学者は指定体系の形而上学的実在を定規にできる
- 自然科学者は経験的観測可能性のscopeでは`unknown`または`not_applicable`とできる

IBDは各AssertionをRegistry、fact scope、World、主張主体、Evidenceとともに返し、別Registryの定規で降格・普遍化・矛盾解消しない。構造上存在しないEvidenceや校正を捏造しないことと、Assertion内容の真偽を裁定しないことを同時に守る。
