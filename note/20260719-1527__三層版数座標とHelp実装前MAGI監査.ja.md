## 考察

三層座標は大小順だけを表す製品版数ではなく、Presentation、Function、SemanticKernelを別責務として
保持する接続座標である。右端一致は陸続きの十分条件ではなく、同一因果次元へ属し得る必要条件である。

```text
SemanticKernel match + World Config compatible + negotiation success
  -> contiguous rendering candidate

SemanticKernel match + World Config incompatible
  -> portal / gate / isolated instance

SemanticKernel mismatch + registered causal gateway
  -> cross-causal projection only

otherwise
  -> bottom / reject
```

異Kernel間ではraw OAE、Agent identity、ASTRO stateを同一物として移送しない。Sourceを不変に保ち、
接続先の因果法則で再構成可能なprojection、incarnation、lineage、receiptを生成する。変換不能部分は
`unknown`または`bottom`として保持する。

Help問題はpersona routingの不足ではなく、persona、習熟度、意図を一つへ潰したcontext事故である。
初期習熟度は`unknown`、初期意図は`look-around`とし、利用者が実装を選ぶまでbranch作成やcode変更を
開始しない。実装済み、足場、未実装、資源待ちも分けて案内する。
